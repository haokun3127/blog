知识速览

概念 作用 复习重点
Tools 执行操作 模型驱动，可能有副作用
Resources 提供数据 应用驱动，只读，无副作用

详细笔记

为什么不能所有东西都用 Tools

Tools 的定位是执行操作，比如查订单、下单、发邮件、提交退货申请。它可以读数据，也可以写数据，甚至触发外部流程。

但有些场景只是读取资料：

场景 如果用 Tools 更合适的方式
读取应用配置 getAppConfig() Resource
查看数据库表结构 getTableSchema(tableName) Resource Template
读取退货政策 getReturnPolicy() Resource
获取系统运行状态 getSystemStatus() Resource

这些操作没有副作用，不应该让模型先推理“我要不要调用工具”。应用完全可以提前决定要读什么资料，再把资料放进上下文。

这就是 Resources 的定位：

注意：MCP 只负责定义资源如何被发现和读取，不负责自动把资源塞进 Prompt。资源读出来以后怎么用，仍然是 Host 应用的编排策略。

Resources 和 Tools 的核心区别

Resources 与 Tools 最容易混，复习时可以从“副作用”和“控制方”两个角度区分。

维度 Tools Resources
本质 执行操作 提供数据
是否有副作用 可能有 无，只读
谁决定调用 模型决定 应用或用户决定
协议方法 tools/call resources/read
典型场景 下订单、发邮件、查年假 读文档、读配置、读表结构
类比 点菜，让厨房做事 看菜单，读取信息

Resources 适合什么场景

Resources 的价值不是“能读数据”，因为 Java 后端本来就能通过 Feign、Dubbo、gRPC、数据库或配置中心读数据。

它真正有价值的场景是：

比如一个 MCP Server 暴露企业知识库资料：

这时资源读取逻辑写在 Server 端一次，所有 Client 通过统一的 resources/read 协议访问，不需要每个 Client 单独写一套数据获取逻辑。

如果只有一个 Java 后端作为唯一 Client，并且已有成熟微服务调用链路，那么 Resources 不是必选项，直接在后端查数据、拼上下文也完全合理。

两种资源类型：直接资源与资源模板

MCP Resources 分两类：Direct Resources 和 Resource Templates。

直接资源

直接资源有固定 URI，指向确定的数据。

它适合数量有限、地址固定、不依赖业务参数的数据，比如系统公告、服务条款、退货政策、应用配置。

资源模板

资源模板的 URI 里带参数，读取时需要填入具体变量。

它适合数量动态、依赖业务参数的数据。比如订单详情不可能每一单都注册一个固定资源，用模板更合理。

怎么选

场景 选择 原因
退货政策 直接资源 地址固定，内容有限
应用配置 直接资源 通常不依赖参数
用户资料 资源模板 依赖 userId
订单详情 资源模板 依赖 orderId
数据库表结构 资源模板 依赖 tableName
日期日志 资源模板 依赖日期和服务名

复习重点

  1. Resources 是只读上下文，不是工具操作。
  2. Resources 的调用通常由 Host 或应用控制，不是模型自己决定。
  3. Direct Resource 适合固定 URI，Resource Template 适合带参数的动态资源。
  4. resources/read 读取的是资源内容，读完以后怎么拼进 Prompt 由 Host 决定。
  5. 资源订阅只通知变化，Client 通常还要再次 resources/read。
  6. Prompts 返回的是 messages 模板，不是自动请求模型。
  7. Prompt 工程解决“怎么写好 Prompt”，MCP Prompts 解决“怎么复用好 Prompt”。
  8. Tools、Resources、Prompts 的选型关键是：是否有副作用、谁来控制、返回什么。

易混点

易混点 正确理解
Resources 是不是工具的一种 不是。Resources 是只读资料，Tools 是执行操作
Resources 会不会自动进入模型上下文 不会。Host 读取后自己决定怎么用
Resources 和后端查数据库有什么区别 单 Client 时区别不大，多 Client 共享时价值明显
Resource Template 是不是接口 它更像带参数的资源 URI 模板
Prompts 是不是 Prompt 工程 不是。Prompt 工程写模板，Prompts 分发模板
prompts/get 会不会直接调用模型 不会，只返回 messages
Tools、Resources、Prompts 谁最重要 三者各司其职，成熟 MCP Server 往往都会提供

面试表达

如果面试官问：MCP 里的 Tools、Resources、Prompts 有什么区别？

可以这样回答：

Tools、Resources、Prompts 是 MCP 暴露能力的三种方式。Tools 负责执行操作，通常由模型根据用户意图决定是否调用，可能有副作用，比如查订单、发邮件、提交申请。Resources 负责提供只读数据，通常由 Host 或应用主动读取，然后作为上下文给模型，比如读取配置、文档、表结构。Prompts 负责提供标准化 Prompt 模板，Client 传入参数后拿到 messages，用来复用团队沉淀的问答模板、摘要模板等。简单说，Tools 是做事，Resources 是给资料，Prompts 是给模板。

如果面试官问：为什么不用 Tools 读取所有数据？

可以这样回答：

不是所有读取都适合做成 Tools。Tools 是模型驱动的，模型要先推理需不需要调用工具，这会引入额外延迟，也有漏调用风险。而很多上下文其实应用一开始就知道应该加载，比如用户资料、权限、配置、政策文档，这些更适合用 Resources，由 Host 主动读取后放进上下文。这样链路更稳定，也能减少模型决策的不确定性。

如果面试官问：MCP Prompts 和普通 Prompt 工程是什么关系？

可以这样回答：

Prompt 工程关注的是如何设计一个效果好的 Prompt，比如角色、约束、引用规则、兜底策略。MCP Prompts 关注的是如何把这些已经验证过的 Prompt 模板标准化地暴露给多个 Client。它不是替代 Prompt 工程，而是把 Prompt 工程的成果封装成可复用模板，通过 prompts/list 和 prompts/get 分发出去。

本节小结

这一节把 MCP 的三类能力补完整了。

第 13 节里 MCP 还只是一个总览概念：Host、Client、Server、Tools、Resources、Prompts。学完这一节后，Resources 和 Prompts 的定位就清晰了：

对 Ragent 这种 Agentic RAG 系统来说，这三者会共同服务于最终问答链路：Resources 提供稳定上下文，Prompts 固化问答规则，Tools 在知识库答不了或需要外部动作时补能力。
