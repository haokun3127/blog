Ragent学习笔记13：MCP 协议入门与实践

学习导览

第 12 小节学了 Function Call：模型可以根据用户问题输出 tool_calls，应用代码执行函数，再把结果交给模型生成答案。

但 Function Call 工具一多，就会出现管理问题：

MCP 要解决的不是“模型会不会调用工具”，而是“工具怎么被标准化管理、发现、连接和调用”。

可以先记一句话：

知识速览

概念 作用 复习重点
MCP Model Context Protocol，模型上下文协议 标准化工具、资源、提示词的连接方式
Host 宿主应用 用户交互入口，比如 Claude Desktop、Cursor、自研助手
Client MCP 客户端 Host 内部连接 Server 的组件
Server MCP 服务端 暴露工具、资源、提示词的一方
Tools 工具调用 查年假、查订单、查物流等可执行能力
Resources 资源访问 读取文件、配置、文档等上下文资源
Prompts 提示词模板 提供标准化 Prompt 模板
Stdio 标准输入输出传输 适合本地工具
Streamable HTTP HTTP + 可流式传输 适合远程服务和团队共享
JSON-RPC MCP 消息基础格式 用 JSON 表示请求、响应、调用 ID

详细笔记

1. Function Call 工具多了会出什么问题

Function Call 解决的是“模型怎么表达调用意图”。但它没有完整解决“工具怎么管理”。

如果只有两个工具，比如查年假、查订单，手写 JSON Schema 和 if-else 路由还能接受。

但如果工具变成几十个：

痛点 表现
工具定义难维护 每个工具都要写 name、description、parameters
Schema 容易不同步 代码参数改了，工具定义忘了改
工具发现困难 新同事不知道有哪些工具
跨语言困难 Java、Python、Node 工具各写各的
权限复杂 哪些用户能调用哪些工具要单独管理
观测困难 调用了哪个工具、耗时多少、失败原因是什么都要自己埋点

Function Call 本身偏“模型调用协议”，MCP 则偏“工具管理协议”。

2. MCP 是什么

MCP 全称 Model Context Protocol，模型上下文协议。

它是一个开放协议，用来标准化大模型应用和外部工具、资源、提示词之间的连接。

可以把 MCP 类比成大模型世界的 USB 接口：

类比 MCP 中对应
电脑 Host 宿主应用
USB 接口 MCP Client
外设 MCP Server
设备能力 Tools / Resources / Prompts
统一连接标准 MCP 协议

没有 MCP 时，每个工具都要为每个应用单独适配。

有 MCP 后，只要 Server 实现 MCP 协议，支持 MCP 的客户端就可以发现和调用它。

3. MCP 的三层架构

MCP 的核心是 Host、Client、Server 三层。

角色 说明 职责
Host 用户直接使用的应用 接收用户输入、调用模型、展示答案
Client Host 内部的连接组件 连接某个 MCP Server，发现工具，发送调用请求
Server 提供能力的一方 暴露工具、资源、提示词，执行调用并返回结果

一个 Host 可以有多个 Client，每个 Client 连接一个 Server。

这点很重要：

比如企业助手可以同时连接：

  • HR Server：查年假、查考勤。
  • 订单 Server：查订单、查物流。
  • 知识库 Server：检索文档。

4. MCP 通信流程

图里的流程可以拆成：

  1. 用户在 Host 中输入问题。
  2. Host 调用 LLM 分析用户意图。
  3. LLM 决定调用工具。
  4. Host 通过对应 MCP Client 向 MCP Server 发起 tools/call。
  5. Server 执行工具并返回结果。
  6. 结果回传给 LLM，LLM 生成最终答案。

这里的重点是：Host 不需要在本地手写每个工具的 JSON Schema。Client 启动后，可以从 Server 动态发现工具列表。

这就解决了 Function Call 里很烦的一点：

5. MCP 的三大能力

MCP 不只支持 Tools，还定义了三类能力。

能力 作用 示例
Tools 执行动作或查询业务系统 查年假、查订单、查天气、创建会议
Resources 暴露可读取资源 文件内容、配置、数据库表结构、文档
Prompts 提供提示词模板 代码审查模板、日报总结模板、客服回答模板

本节重点是 Tools，因为它和 Function Call 关系最直接。

Tools 和 Function Call 的区别：

对比 Function Call MCP Tools
工具定义位置 通常写在应用端请求里 Server 端定义，Client 动态获取
工具发现 应用代码自己维护 协议支持发现
跨语言 需要自己适配 协议统一，语言无关
管理能力 较弱 更适合权限、监控、注册和运维

6. MCP 和 Function Call 的关系

这张图很关键。

Function Call 是大模型的原生能力，提供：

  • 意图识别。
  • 判断是否调用工具。
  • 输出 tool_calls JSON。
  • 多轮对话中使用工具结果生成答案。

MCP 在 Function Call 上层，提供：

  • 工具注册与发现。
  • 标准化传输。
  • 权限与安全控制。
  • 多语言 SDK。
  • 跨语言、跨平台支持。

所以 MCP 不是替代 Function Call，而是增强 Function Call。

7. MCP 协议和实现框架不要混淆

课程里提到 Spring AI MCP Server 可以用 @Tool 注解定义工具。但这不是 MCP 协议本身的全部。

要区分两层：

层次 作用
MCP 协议 定义工具元数据格式、通信方式、工具发现和调用规范
Spring AI 等实现框架 帮开发者更方便地实现 MCP Server

8. MCP 传输方式

MCP 主要有两种传输方式：Stdio 和 Streamable HTTP。

8.1 Stdio

Stdio 是标准输入输出。

Client 启动一个本地 Server 子进程，然后通过 stdin/stdout 传递 JSON-RPC 消息。

流程：

适合：

  • 本地文件工具。
  • 本地数据库工具。
  • 个人开发工具。
  • Claude Desktop、Cursor 调用本地工具。

限制：

  • Server 和 Client 通常在同一台机器。
  • 不适合多人共享。
  • 远程部署能力弱。

8.2 Streamable HTTP

Streamable HTTP 是基于 HTTP 的传输方式，适合远程服务。

流程：

适合：

  • 企业内部工具平台。
  • 多个客户端共享同一个 Server。
  • 远程服务。
  • 需要接入认证、监控、负载均衡的场景。

对比：

传输方式 适合场景 优点 缺点
Stdio 本地工具、个人开发 简单、启动快、无需网络 不适合远程共享
Streamable HTTP 企业服务、团队共享 支持远程、HTTP 生态成熟、可流式 需要网络配置、认证和部署

  1. 调 HR Server 查年假。
  2. 调日历 Server 查会议。
  3. 收集两个结果。
  4. 交给 LLM 生成综合答案。

MCP 的结构天然适合这种多工具、多 Server 的组合：

10. MCP 的工具发现

Function Call 里，工具列表通常是应用端手动写进请求。

MCP 里，工具定义在 Server 端。Client 连接 Server 后，可以请求工具列表。

工具元数据通常包括：

元数据 作用
工具名 用于调用
工具描述 给模型判断是否使用
参数 Schema 说明参数名、类型、必填项
返回格式 说明工具结果结构

11. MCP 的安全边界

MCP 让工具更容易被发现和调用，也意味着安全边界更重要。

需要关注：

风险 处理方式
越权调用 Server 端做身份认证和权限判断
敏感数据泄露 控制工具返回字段，必要时脱敏
危险操作 删除、付款、发送消息要二次确认
工具滥用 限流、审计、调用频率监控
参数注入 参数校验和白名单
跨服务调用失败 超时、重试、降级和错误说明

不要把安全判断交给模型。模型可以建议调用工具，但真正的权限控制必须在 Server 或业务系统里完成。

复习重点

  • Function Call 解决“模型输出工具调用意图”，MCP 解决“工具规模化管理和标准化连接”。
  • MCP 三层架构是 Host、Client、Server。
  • 一个 Host 可以有多个 Client，一个 Client 通常连接一个 Server。
  • MCP 的三大能力是 Tools、Resources、Prompts，本节重点是 Tools。
  • MCP 不替代 Function Call，而是 Function Call 之上的标准化封装层。
  • Stdio 适合本地工具，Streamable HTTP 适合远程服务和团队共享。
  • 工具发现是 MCP 的关键价值：工具定义在 Server 端，Client 动态获取。
  • 权限、安全、监控仍然必须由 Server 和业务系统兜住。

易混点整理

易混点 正确理解
MCP 替代 Function Call 不对。MCP 依赖模型的 Function Call 能力
MCP 就是 Spring AI 的 @Tool 注解 不对。注解只是实现框架提供的便利方式
Host、Client、Server 是同一个东西 不对。Host 是应用，Client 是连接组件，Server 是能力提供方
一个 Client 可以随便连多个 Server 通常一个 Client 连接一个 Server，一个 Host 有多个 Client
Stdio 和 HTTP 只是写法不同 不只是写法，部署方式、适用场景和运维方式都不同
MCP 有了就不用做权限控制 不对。权限、安全、审计仍要工程实现

本节小结

第 13 节把工具调用从“单个工具怎么调”推进到了“很多工具怎么管理”。

Function Call 让模型知道“我应该调用什么工具”；MCP 让系统知道“工具在哪里、怎么发现、怎么调用、怎么跨语言复用”。
