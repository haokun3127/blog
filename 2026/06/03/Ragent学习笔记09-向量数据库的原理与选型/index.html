Ragent学习笔记09：向量数据库的原理与选型

发表于2026-06-03|更新于2026-06-03|Ragent

  • 《AI大模型Ragent项目》第9小节：向量数据库的原理与选型

学习导览

第 8 小节已经把文本 Chunk 转成了 Embedding 向量，也理解了“用语义相似度找内容”的基本流程。第 9 小节继续往下问一个更工程化的问题：

如果只是 Demo，把向量放进 List<float[]> 或 MySQL 字段里，再逐条计算余弦相似度也能跑通。但真实 RAG 知识库里，Chunk 数量可能是几十万、几百万甚至更多。每次用户提问都全量遍历，延迟、CPU、内存、I/O 都会顶不住。

所以这一节的核心不是“怎么把向量存起来”，而是“怎么在海量向量里快速找到最相似的 TopK”。这就是向量数据库的价值：向量存储、ANN 索引、高效检索和元数据过滤。

知识速览

概念 作用 复习重点
暴力搜索 逐个计算 Query 向量和所有向量的相似度 精确但慢，数据量大时不可用
ANN Approximate Nearest Neighbor，近似最近邻搜索 用少量召回损失换取巨大性能提升
IVF 先聚类分区，再只搜索相关簇 关注 nlist 和 nprobe
HNSW 分层图结构，从粗到细查找近邻 速度快、召回高，但内存占用大
FLAT 不建 ANN 索引，直接精确搜索 适合小数据量或极高精度要求
Milvus 专用向量数据库 支持向量索引、标量过滤、分区和 Java SDK
Collection Milvus 中的数据集合 类似 MySQL 的表
Schema Collection 字段结构 包含向量字段和标量字段
Index 索引 向量字段建 ANN 索引，标量字段建过滤索引
Partition 分区 按业务维度缩小搜索范围

详细笔记

1. 为什么普通数据库不够用

向量本质上是一组浮点数数组。最直觉的做法是：在 MySQL 里建一个字段，把向量序列化成 JSON 或字符串存进去。

简化结构可以理解为：

这样做能不能存？能。问题出在检索。

用户提问后，系统要先把用户问题也转成 Query 向量，然后拿 Query 向量和库里的每个 Chunk 向量逐个计算相似度，最后排序取 TopK。

如果知识库只有几十条 Chunk，这样没问题。但假设有 100 万个 Chunk，每个向量是 4096 维，每次查询都要：

  1. 读出 100 万个向量。
  2. 每个向量都和 Query 向量计算一次余弦相似度。
  3. 对 100 万个相似度分数排序。
  4. 返回 TopK。

这就是暴力搜索。它的问题不是不能用，而是数据量和并发量上来之后完全扛不住。

这张图很直观：每次查询都要遍历全部向量，时间复杂度是 O(n)。如果库里有 100 万条向量，单次查询可能就要几秒。RAG 问答还要经历 Embedding、检索、重排、Prompt 拼装、LLM 生成等步骤，向量检索单独耗几秒就已经不可接受。

所以普通数据库的问题是：

问题 说明
能存但不好查 MySQL 可以保存向量，但没有高效 ANN 检索能力
全量遍历成本高 每次查询都要读出大量向量并逐个计算
排序开销大 相似度结果越多，排序成本越高
并发不友好 多个用户同时查询时 CPU 和 I/O 压力明显
元数据过滤和向量检索割裂 需要应用层自己组合过滤与相似度计算

结论：普通数据库适合存结构化数据，不适合承担大规模向量相似度检索的主力工作。

2. ANN：用近似结果换检索速度

ANN 是 Approximate Nearest Neighbor，近似最近邻搜索。

它的核心思想是：

这听起来像“牺牲准确率”，但在 RAG 场景里是合理的。因为 RAG 通常不会只拿 Top1，而是拿 Top5、Top10、Top20 作为候选，再经过元数据过滤、重排和 Prompt 约束。只要相关内容进入候选集，后续链路还有机会修正排序。

可以这样理解：

检索方式 速度 召回 适合场景
暴力搜索 慢 最高 小数据量、验证 Demo、极高精度场景
ANN 检索 快 略有损失 大多数生产 RAG 场景

向量数据库真正解决的不是“保存向量”，而是“通过 ANN 索引快速召回相似向量”。

3. IVF：先分区，再搜索相关区域

IVF 的全称是 Inverted File Index，倒排文件索引。

它的核心思路是：

建索引阶段：

  1. 用 K-Means 之类的聚类算法，把所有向量分成 nlist 个簇。
  2. 每个簇有一个中心点。
  3. 每条向量归到离自己最近的簇里。

查询阶段：

  1. Query 向量先和各个簇中心点计算距离。
  2. 选出最近的 nprobe 个簇。
  3. 只在这些簇内部继续搜索。

图里的 nlist=10 表示总共分 10 个簇，nprobe=2 表示查询时只搜 2 个簇。这样搜索范围从 100 万向量缩小到大约 20 万向量，计算量明显下降。

IVF 的关键参数：

参数 含义 调大影响 调小影响
nlist 聚类簇数量 每个簇更小，检索更快，但训练更慢 每个簇更大，检索更慢
nprobe 查询时搜索多少个簇 召回率更高，检索更慢 检索更快，但可能漏召回

IVF 的优点是原理简单、内存占用相对低，适合数据量很大的场景。缺点是对聚类效果敏感，靠近簇边界的向量可能被漏掉，所以需要通过 nprobe 调参平衡速度和召回。

常见 IVF 变体：

索引类型 特点 适用场景
IVF_FLAT 簇内仍做精确搜索 数据量较大，但希望保持较好召回
IVF_SQ8 对向量做标量量化压缩 内存有限，需要降低存储成本
IVF_PQ 乘积量化，压缩更激进 超大规模数据，对内存特别敏感

4. HNSW：用分层图快速找近邻

HNSW 的全称是 Hierarchical Navigable Small World Graph，分层可导航小世界图。

它可以理解成一种“从粗到细”的图搜索结构：

  • 顶层节点少，连接跨度大，用来快速定位大概区域。
  • 中间层节点更多，范围更细。
  • 底层包含所有向量，用来做最终精确逼近。

图里把 HNSW 类比成“问路找餐厅”：

  1. 先问熟悉全局的人，确定去哪个城市。
  2. 再问熟悉局部的人，确定哪个区。
  3. 最后问具体街道，定位到目标位置。

放到向量检索里，就是从顶层稀疏图开始跳转，逐层下降，到底层找到最接近 Query 的向量。

HNSW 为什么快：

原因 说明
分层结构 顶层快速粗定位，底层精细搜索
图连接 每个节点连接若干近邻，可以沿着更接近 Query 的方向移动
避免全量遍历 不需要和所有向量都计算距离
召回效果好 在很多场景下召回率和速度都比较均衡

HNSW 的代价是内存。因为它不仅要存向量本身，还要存图结构里的边。向量数量越多、维度越高、连接数越大，内存压力越明显。

HNSW 关键参数：

参数 含义 调大影响 调小影响
M 每个节点的最大连接数 召回率提升，内存增加，建索引变慢 内存降低，但召回可能下降
efConstruction 建索引时的搜索宽度 索引质量提升，构建变慢 构建更快，但索引质量下降
ef 查询时的搜索宽度 召回提升，查询变慢 查询更快，但可能漏召回

5. 索引类型怎么选

向量索引没有绝对最优，只有适不适合当前场景。

数据规模 推荐方向 说明
小于 10 万 FLAT 直接暴力搜索也能接受，少调参
10 万到 500 万 HNSW 多数 RAG 项目的默认选择，速度和召回较均衡
500 万到 5000 万 HNSW 或 IVF_SQ8 内存够用 HNSW，内存紧张考虑压缩
5000 万以上 DISKANN 或 IVF_PQ 索引放磁盘或更强压缩，适合超大规模

常见索引对比：

索引 检索速度 召回率 内存占用 适合场景
FLAT 慢 最高 低 小数据量、精确检索
IVF_FLAT 中 中高 中 数据量大，想节省检索成本
IVF_SQ8 中快 中 低 内存有限，需要压缩
HNSW 快 高 高 大多数生产 RAG 场景
DISKANN 中快 高 低 数据量极大，内存放不下

6. 向量数据库怎么选

向量数据库方案大致分两类。

第一类是专用向量数据库，比如 Milvus、Qdrant、Weaviate、Pinecone、Chroma。它们从设计上就是为向量检索服务的，向量是一等公民，ANN 索引、标量过滤、批量插入、索引构建都更专业。

第二类是传统数据库的向量扩展，比如 PostgreSQL 的 PgVector、Elasticsearch 的 kNN、MySQL 的向量能力。它们的优势是基础设施复用，不需要额外维护一套数据库，但在超大规模向量检索、索引类型丰富度和专项优化上通常不如专用向量数据库。

方案 优点 代价 适合场景
Milvus 索引类型丰富、Java SDK 完善、适合大规模向量检索 需要单独运维 百万级以上向量、对检索性能要求高
PgVector 复用 PostgreSQL，工程集成简单 超大规模性能和索引能力相对有限 中小规模知识库、已有 PostgreSQL 项目
Elasticsearch + kNN 全文检索强，生态成熟 混合检索和向量专项能力要看版本与配置 已有 ES 基础设施，关键词检索要求高
Chroma 轻量，原型验证方便 生产级能力有限 学习、Demo、Python 原型
Pinecone 托管省运维 成本和数据控制要评估 希望快速上线且接受托管服务

课程这一节用 Milvus 帮助理解向量数据库。即使后续 Ragent 项目切换到 PgVector，这一节仍然很有价值，因为索引、召回、元数据过滤、TopK、相似度度量这些概念是通用的。Milvus 更适合理解“专用向量库”的完整能力，PgVector 更偏向“和业务数据库合并后的工程取舍”。

7. Milvus 核心概念

Milvus 的概念可以和 MySQL 类比理解。

这张图把 Milvus 的几个核心组件串起来了：

Milvus 概念 类比 MySQL 作用
Collection 表 存储一类向量数据
Schema 表结构 定义字段，比如 id、vector、chunk_text、doc_id、category
Vector Field 无直接等价 存 Embedding 向量，用于相似度检索
Scalar Field 普通列 存元数据，用于过滤、展示和追踪
Index 索引 向量字段建 ANN 索引，标量字段建过滤索引
Partition 分区 按业务维度拆分数据，缩小检索范围

典型 RAG Collection 里不会只存向量，还会存完整的 Chunk 记录：

字段 示例 用途
id 自动生成主键 唯一标识
vector 4096 维浮点数组 做向量相似度检索
chunk_text 退货政策正文片段 后续拼 Prompt
doc_id doc_return_001 关联源文档，支持更新删除
category return_policy 标量过滤或分区

8. 元数据过滤和 Partition

RAG 检索不能只看语义相似度，还要看“这条结果能不能给当前用户看”“是不是当前知识库”“是不是当前业务分类”。

常见过滤条件：

过滤维度 示例 作用
知识库 kb_id = customer_service 限定搜索范围
文档状态 status = published 避免召回草稿或失效文档
权限 department = after_sales 避免越权
分类 category = return_policy 提高结果相关性
时间 effective_date <= now 避免过期政策

Partition 和标量过滤都能缩小范围，但使用方式不同：

方式 适合情况 注意点
Partition 某个过滤条件几乎每次都用，且取值不多 Partition 太多会增加管理复杂度
标量过滤 过滤条件灵活、取值多、经常组合查询 数据量特别大时性能要压测

9. 相似度度量怎么选

Milvus 支持多种度量方式，文本 RAG 里最常见的是 COSINE。

度量方式 含义 越大越相似 适合场景
COSINE 看向量方向夹角 是 文本语义检索最常见
IP 内积 是 向量已归一化时和 COSINE 接近
L2 欧氏距离 否，越小越近 图像、音频、推荐等场景

10. 数据更新策略

知识库文档会新增、修改、删除，所以向量库也要同步变化。

Milvus 里更新向量通常不是“原地改字段”，而是按文档维度删旧插新：

这再次说明 doc_id、chunk_id、embedding_model、embedding_dimension 这些元数据很重要。没有这些字段，后续排查、重建、删除、灰度切换都会很麻烦。

复习重点

  • 向量数据库的重点不是“能不能存向量”，而是“能不能高效做近似最近邻检索”。
  • 暴力搜索精确但慢，ANN 检索用可接受的召回损失换取毫秒级性能。
  • IVF 是先聚类分区，再搜索相关簇；核心参数是 nlist 和 nprobe。
  • HNSW 是分层图搜索，速度和召回都很好，但内存占用较高；核心参数是 M、efConstruction、ef。
  • Milvus 的 Collection、Schema、Index、Partition 可以类比 MySQL 的表、表结构、索引、分区。
  • RAG 检索一定要把向量相似度和元数据过滤结合起来，否则容易召回越权、过期或错误分类的内容。
  • Milvus 和 PgVector 不是谁绝对更好，而是取舍不同：Milvus 偏大规模向量检索能力，PgVector 偏工程集成和运维简化。

易混点整理

易混点 正确理解
能存向量就是向量数据库 不对。关键是 ANN 索引和高效相似度检索
ANN 是“不准确”的坏方案 不对。ANN 是工程上用速度换少量召回损失的合理方案
HNSW 一定比 IVF 好 不一定。HNSW 召回和速度好，但内存压力更高
Partition 越多越好 不对。Partition 太多会增加管理复杂度
换 Embedding 模型只要维度一样就不用重建 不对。语义空间可能不同，通常要重新向量化
TopK 越大越好 不一定。过大的 TopK 会增加重排和 Prompt 成本
