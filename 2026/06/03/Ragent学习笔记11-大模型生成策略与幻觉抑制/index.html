学习导览

第 10 小节已经解决了“怎么检索得更准”：混合检索召回候选，RRF 做融合，Rerank 把真正相关的 Chunk 排到前面。

但 RAG 不是检索结束就万事大吉。即使 TopK Chunk 召回正确，大模型仍然可能答错。原因是模型会用自己的训练知识补全、改写甚至覆盖检索结果。

第 11 小节解决的是生成阶段的最后一公里：

这一节的关键不是写一个“请根据资料回答”的简单 Prompt，而是建立一套生成约束：系统指令、检索上下文、用户问题、引用规则、兜底策略、输出格式和参数控制。

知识速览

概念 作用 复习重点
生成阶段 基于检索结果生成答案 检索对了，生成仍可能错
幻觉 模型生成看似合理但不真实的内容 RAG 中常表现为无依据、篡改或拼错信息
System Prompt 定义角色、边界和规则 要明确“只基于参考资料回答”
Retrieved Context 检索得到的 TopK Chunk 需要编号、来源、时间等元数据
User Query 用户真实问题 和上下文一起传给模型
引用对齐 答案标注来源编号 让答案可追溯、可验证
兜底回答 资料不足时明确说无法回答 幻觉抑制的最后防线
Temperature 控制生成随机性 RAG 场景通常设低一些
输出约束 控制答案格式、长度、边界 可用 JSON、列表、简洁回答等形式

所以 RAG 的质量不是只靠检索，还要靠生成约束。检索负责把证据找出来，Prompt 负责让模型“按证据说话”。

2. RAG 生成阶段的三个核心挑战

挑战 表现 风险
幻觉 编造资料中没有的信息 用户拿到错误答案
答案不可追溯 没有引用来源 用户无法验证答案依据
输出不稳定 长短、格式、边界不一致 下游展示和程序解析困难

其中幻觉是最关键的问题。

在 RAG 场景里，用户默认以为答案来自知识库。如果模型偷偷掺入自己的训练知识，答案看起来像“企业知识库回答”，实际却可能没有依据，风险比普通闲聊更大。

Retrieved Context 给模型看的参考资料 工作资料
User Query 用户实际问题 客户提问

4. System Prompt 应该约束什么

System Prompt 不是越长越好，而是要清晰规定模型不能做什么、必须做什么。

RAG 场景常用规则：

规则 目的
只基于参考资料回答 防止模型用训练知识自由发挥
资料不足时明确说明无法回答 防止硬编
不编造数字、日期、金额、流程细节 这些是幻觉高发区
回答时标注引用编号 让答案可追溯
多条资料冲突时指出冲突 避免自行合并出错误结论
用简洁、友好的语气回答 控制输出风格

6. 上下文窗口不是越大越好

很多模型支持很长上下文，但 RAG 里不能简单地把所有 Chunk 都塞进去。

问题有三个：

问题 说明
成本增加 输入 Token 越多，费用越高
延迟增加 模型处理长上下文更慢
噪声增加 无关 Chunk 越多，模型越容易被干扰

还有一个常见现象：模型对上下文中间位置的信息关注较弱，关键信息如果被大量内容淹没，反而可能被忽略。

所以第 10 节的 Rerank 很重要。它的目标就是把真正相关的 Chunk 排到前面，让进入 Prompt 的上下文更干净。

7. 幻觉的三种典型表现

类型 示例 问题
无依据编造 资料没写运费，模型回答“运费由商家承担” 资料外补全
篡改事实 资料写“不支持退货”，模型回答“可以退货” 用常识覆盖资料
错误组合 A 资料说 7 天可退，B 资料说定制商品不可退，模型合并成“定制商品 7 天可退” 把局部事实拼成错误结论

RAG 幻觉最危险的地方是：它往往非常流畅，用户不容易立刻发现。

8. 幻觉抑制的实用技巧

技巧 写法或做法 作用
限定知识来源 “只基于参考资料回答” 防止模型自由发挥
明确兜底 “资料不足时请说明无法回答” 防止硬编
禁止编造具体信息 “不要编造数字、日期、金额、政策细节” 降低高风险幻觉
要求引用 “回答时标注 [1]、[2]” 让模型为答案找证据
控制参数 低 Temperature 减少随机发挥
控制上下文 只给高质量 TopK 减少噪声干扰

9. Temperature 和 Top-P

调用大模型 API 时，temperature 会影响生成随机性。

Temperature 直觉 适合场景
0 最保守，输出最确定 结构化、严谨问答
0.1 - 0.3 稳定但不僵硬 RAG 客服、知识库问答
0.7 有明显随机性 普通聊天、文案
1.0+ 更发散 创意写作、头脑风暴

RAG 追求准确，不追求创意，所以通常设置低温度。

课程里提到一个实用选择：

不一定非要设成 0，因为完全确定的输出有时会生硬；0.1 保留一点点表达灵活性，同时基本不鼓励模型乱编。

top_p 也能控制采样范围，但一般不建议一开始就同时调 temperature 和 top_p。先调一个变量，更容易观察效果。

11. 输出格式和边界约束

RAG 不一定只输出普通自然语言。不同场景需要不同格式。

输出形式 适用场景 Prompt 约束
简洁文本 客服问答 “用 2-3 句话回答核心结论”
列表 操作步骤、注意事项 “按条列出步骤”
JSON API 接口、下游程序消费 “请输出 JSON，包含 answer、citations、confidence”
带引用文本 知识库问答 “每个关键结论后标注引用编号”

如果要求 JSON，最好给出字段示例，否则模型可能自己发明字段名。

复习重点

  • RAG 不是检索到 Chunk 就结束，生成阶段仍然可能幻觉。
  • Prompt 的核心作用是让模型把检索结果当作唯一依据，而不是普通参考。
  • RAG Prompt 通常由 System Prompt、Retrieved Context、User Query 三段组成。
  • Retrieved Context 最好带编号、来源、更新时间等元数据，方便引用对齐。
  • 幻觉抑制要靠多层手段：只基于资料、资料不足就兜底、禁止编造细节、引用编号、低 Temperature。
  • 引用对齐不仅是展示优化，也是排错和审计能力。
  • 上下文不是越多越好，TopK 要控制噪声、成本和延迟。

易混点整理

易混点 正确理解
检索正确就一定回答正确 不一定。模型可能忽略、改写或覆盖检索内容
Prompt 写“根据资料回答”就够了 不够。还要明确资料是唯一依据，以及资料不足时怎么处理
幻觉可以完全消除 不能完全消除，只能通过工程策略降低概率
Temperature 越高回答越好 RAG 场景通常相反，低温度更稳定
引用只是给用户看的装饰 不对。引用还用于验证、排错、审计
上下文塞越多越保险 不一定。无关上下文会增加噪声和成本
