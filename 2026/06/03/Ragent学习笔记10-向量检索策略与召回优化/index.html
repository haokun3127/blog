学习导览

第 9 小节已经能用向量数据库做语义检索了：用户问题转成 Query 向量，到 Milvus 或 PgVector 里找相似 Chunk。

但真实 RAG 系统里，“能检索”不等于“检索得准”。纯向量检索擅长理解语义，却不擅长处理订单号、商品型号、年份、专有名词、缩写这类精确关键词。

第 10 小节的重点就是从“向量召回”升级到“检索策略”：

这一节可以看成 RAG 检索质量的分水岭。只会向量检索，容易做出 Demo；理解混合检索和重排序，才更接近生产可用。

知识速览

概念 作用 复习重点
纯向量检索 按语义相似度召回 Chunk 语义强，但精确关键词弱
BM25 经典关键词检索算法 词频、逆文档频率、文档长度归一化
Hybrid Search 向量检索 + 关键词检索 同时兼顾语义和精确匹配
Dense Vector 稠密向量 Embedding 生成，表达语义
Sparse Vector 稀疏向量 BM25/全文检索生成，表达关键词
RRF Reciprocal Rank Fusion，倒数排名融合 不直接融合分数，而是融合排名
Rerank 重排序 对候选结果做精排，提高最终 TopK 质量
Bi-Encoder 双编码器 查询和文档分别编码，速度快，适合召回
Cross-Encoder 交叉编码器 查询和文档一起输入，精度高，适合重排
TopK 最终给 LLM 的结果数量 太少会漏信息，太多会增加噪声

详细笔记

1. 纯向量检索的短板

纯向量检索把文本转成 Embedding，然后比较语义相似度。它的优势是可以处理同义表达。

比如：

用户问法 文档写法 向量检索表现
买了一周还能退吗？ 自签收之日起 7 天内支持无理由退货 能理解语义接近
新员工多久转正？ 试用期为 3 个月 能理解“转正”和“试用期”有关
物流什么时候发？ 普通商品 48 小时内发货 能按语义召回

但它对精确关键词不敏感。典型问题有三类。

第一类是数字和编号：

向量检索可能把它理解成“物流状态查询”，召回一堆物流规则，却没有优先命中包含 2026012345 的记录。

第二类是商品型号：

向量检索可能抓住“退货政策”这个语义，但忽略 iPhone 16 Pro Max 和“拆封”这两个关键约束。

第三类是专有名词和缩写：

如果模型没有很好理解 RMA，它可能只召回通用退货流程，而不是专门讲 RMA 的文档。

维度 向量检索 关键词检索
核心能力 语义理解 字面精确匹配
擅长 同义词、自然语言问法、意图匹配 编号、型号、年份、专有名词、缩写

两者不是替代关系，而是互补关系。

3. BM25 是什么

BM25 是经典关键词检索算法，用来判断查询词和文档之间的相关性。它不是靠大模型理解语义，而是基于统计信息打分。

BM25 主要看三个因素。

3.1 词频 TF

查询词在文档中出现得越多，文档可能越相关。

但 BM25 不会让词频无限增长。一个词出现 10 次比出现 1 次更重要，但出现 100 次不一定比出现 10 次重要 10 倍。否则长文档或重复堆词会占便宜。

3.2 逆文档频率 IDF

越稀有的词越有区分度。

比如“退货”在售后知识库里可能很多文档都有，区分度不高；但 2026012345、iPhone 16 Pro Max、RMA 这种词出现次数少，一旦命中，相关性往往很强。

4. 混合检索的基本流程

混合检索就是把向量检索和关键词检索同时跑起来，然后融合结果。

图里的流程可以拆成四步：

  1. 用户输入 Query，比如“退货流程”。
  2. 向量检索分支把 Query 做 Embedding 编码，然后到向量库里做 ANN 搜索，拿到语义相似结果。
  3. BM25 检索分支对 Query 做分词和关键词匹配，拿到词汇维度结果。
  4. 两路结果进入 RRF 融合，得到综合 TopK。

这样做的好处是：

分支 能补什么
向量检索 同义词、语义相近、自然语言表达
BM25 检索 精确关键词、数字编号、型号、缩写
RRF 融合 避免不同检索器分数不可比的问题

5. Milvus 原生混合检索 vs ES + Milvus

混合检索有两种常见架构。

方案 做法 优点 代价
Milvus 原生混合检索 Milvus 同时做 Dense 向量检索和 BM25 Sparse 检索 架构简单，不需要双写，API 支持 RRF 全文检索能力相对 ES 基础
ES + Milvus ES 做关键词检索，Milvus 做向量检索，应用层融合 ES 分词、同义词、布尔查询能力强 两套系统、双写一致性、运维复杂度更高

选型可以这样判断：

场景 推荐
大多数 RAG 知识库 优先 Milvus 原生混合检索，简单直接
已经有成熟 ES 基础设施 可以考虑 ES + Milvus
中文分词、同义词、自定义检索语法要求很高 ES + Milvus 更有发挥空间
希望快速跑通项目 Milvus 原生方案更省事

这一节课程使用 Milvus 原生方案，是为了降低环境复杂度，让注意力集中在检索策略本身。

6. 为什么不能直接把两路分数相加

向量检索返回的可能是余弦相似度，比如 0.84。

BM25 返回的是关键词相关性分数，比如 12.3。

这两个分数不在同一个尺度上，不能直接相加：

这样会让 BM25 分数天然压过向量分数，并不代表文档真的更相关。

即使做归一化，也会遇到问题：

问题 说明
分布不同 向量分数可能集中在 0.7 到 0.9，BM25 分数可能跨度很大
归一化不稳定 每次候选集不同，归一化结果也会变
极值影响 某个异常高分可能压缩其他结果差异

所以工程里常用一种更稳的方式：不融合原始分数，而是融合排名。

RRF 的好处：

不依赖原始分数 避免余弦分数和 BM25 分数不可比
实现简单 只需要知道候选排名
稳定 对分数尺度变化不敏感
适合多路检索 不只两路，后面也可以扩展到多通道召回

8. 混合检索为什么还需要 Rerank

混合检索负责把可能相关的候选找出来，但候选排序不一定足够精确。

RAG 最终给 LLM 的上下文窗口很小，可能只放 Top5。也就是说，前 5 条的质量会直接影响答案质量。

如果第 1 条是不相关内容，LLM 很容易被带偏；如果关键 Chunk 被排到第 20 条，没有进入 Prompt，后续生成也救不回来。

所以要区分两个阶段：

阶段 目标 常见数量
召回 Retrieve 别漏掉相关候选 Top20、Top50
重排 Rerank 把最相关的排到最前 Top3、Top5

这张图很关键，它把 RAG 检索看成一个漏斗：

这个漏斗说明了几个点：

阶段 作用 关注点
Retriever 从海量数据里快速捞候选 召回率、延迟、成本
Reranker 从候选中选最相关 排序精度、TopK 质量
LLM 基于上下文生成答案 上下文质量、引用、幻觉抑制

如果召回阶段没捞到关键内容，重排和生成都没法补救。如果召回太多无关内容，Rerank 成本会高，Prompt 噪声也会增加。

所以检索策略不是“TopK 设大一点就行”，而是要控制漏斗每一层的输入输出。

11. 完整链路：Embed -> Retrieve -> Rerank -> Generate

完整在线问答链路可以拆成：

  1. Embed：用户 Query 先向量化。
  2. Hybrid Retriever：同时走 Dense 语义召回和 Sparse/BM25 关键词召回。
  3. Milvus 或向量数据库：返回候选集，比如 Top30。
  4. Reranker：对候选集做精排，选出 Top5。
  5. LLM：把 Top5 Chunk 作为上下文，生成答案和引用。

这条链路比“用户问题 -> 向量检索 -> LLM”更完整，也更接近生产 RAG。

12. 检索参数怎么调

课程里给了一组比较实用的起始参数，可以作为复习和实战的默认思路。

参数 建议起始值 作用 调优方向
Dense topK 20 向量召回候选数 漏召回就加大，延迟高就减小
Sparse topK 20 BM25 召回候选数 Query 精确词多时可适当加大
RRF k 60 排名融合平滑参数 通常先保持默认
Rerank 候选数 30 送入重排的候选数 精度不够可升到 50，但要看延迟
最终 TopK 5 给 LLM 的 Chunk 数量 结合上下文窗口和噪声控制

一个重要调优顺序：

不要一上来就调 Rerank。如果召回阶段已经漏掉关键 Chunk，后面重排再强也没用。

13. 评估指标怎么理解

检索效果不能只靠感觉，需要指标。

指标 含义 关注问题
Recall@20 Top20 候选里是否包含相关 Chunk 有没有漏召回
MRR 第一个相关结果的排名倒数 正确结果是否排得靠前
nDCG@10 排名前 10 的整体排序质量 多个相关结果的排序是否合理
人工标注准确率 人看答案是否正确 最终效果
用户追问率 用户是否继续追问或纠错 回答是否满足需求
答非所问率 答案是否偏题 检索和生成是否被带偏

复习时可以按阶段记：

复习重点

  • 纯向量检索不是关键词检索的替代品，它语义强，但对精确关键词弱。
  • BM25 不理解语义，但对编号、型号、年份、缩写、专有名词很敏感。
  • 混合检索把 Dense 向量召回和 Sparse/BM25 召回结合起来，是生产 RAG 的常见方案。
  • RRF 解决的是“不同检索器分数不可比”的问题，所以它融合排名而不是直接加分。
  • Rerank 不是为了扩大召回，而是为了提高最终进入 Prompt 的 TopK 质量。
  • Bi-Encoder 快，适合全库召回；Cross-Encoder 准，适合候选集重排。
  • 检索调优要先看召回有没有漏，再看排序准不准，最后看生成答案质量。

易混点整理

易混点 正确理解
向量检索能解决所有搜索问题 不对。编号、型号、年份等精确词需要关键词检索补充
BM25 比向量检索落后 不对。BM25 在精确匹配上仍然很重要
混合检索就是两个分数相加 不对。分数尺度不同，常用 RRF 这种排名融合
Rerank 能补救漏召回 不对。没被召回的 Chunk 不会进入重排
TopK 越大越保险 不一定。TopK 太大会增加 Prompt 噪声和 LLM 成本
Cross-Encoder 可以直接扫全库 不现实。它精度高但慢，适合对候选集精排
