概念 作用 复习重点
Function Call 让模型输出工具调用意图 模型只负责判断和生成 JSON
tools 提供给模型的工具列表 包含工具名、描述、参数 Schema

tool_call_id 工具调用 ID 第二轮把结果返回给模型时要对应
两轮对话 先让模型决定工具，再把结果交给模型生成答案 Function Call 的标准流程

详细笔记

1. RAG 的能力边界

RAG 能回答“知识库里有什么”，但不能直接回答“业务系统当前状态是什么”。

用户问题 知识库能否解决 原因
退货政策是什么？ 能 文档里有政策
iPhone 拆封后能退吗？ 能 文档里有商品规则
订单 #12345 到哪了？ 不能只靠文档 需要查实时订单系统
我还剩几天年假？ 不能只靠文档 需要查 HR 系统
帮我创建一个会议 不能只靠文档 需要执行操作

所以 RAG 解决的是“知识问答”，Function Call 解决的是“工具调用”。

第一种是在 Prompt 里写死：

第二种是用规则匹配：

规则方案的问题是表达太多样，规则很快失控。用户可能说“快递到哪了”“我的包裹现在在哪”“订单 12345 怎么还没到”，写硬编码规则不灵活。

Function Call 的价值就是让模型基于自然语言理解用户意图，再输出标准化调用请求。

6. parameters 和 JSON Schema

Function Call 的参数使用 JSON Schema 定义。

常见字段：

字段 含义
type 参数类型，比如 object、string、number、boolean
properties 参数属性定义
description 参数含义
required 必填参数列表
enum 枚举可选值

示例：

参数 Schema 的作用有两个：

  1. 告诉模型该怎么组织参数。
  2. 给后端做参数校验提供依据。

不过后端不能完全信任模型输出。模型生成的参数仍然可能缺失、类型错误或格式不合法，所以应用代码必须二次校验。

7. tool_choice 怎么理解

tool_choice 用来控制模型是否调用工具。

值 含义 适用场景
auto 模型自己判断是否调用 最常用
none 禁止调用工具，只生成文本 只想做普通问答
required 必须调用某个工具或必须调用工具 明确需要工具时
指定工具 强制调用某个函数 用户已经选择了具体操作

大多数 Agent/RAG 场景用 auto，让模型自己判断。

但强制调用工具要谨慎。比如用户只是问“年假制度是什么”，不一定需要查个人年假余额。

8. 响应中的 tool_calls

如果模型决定调用工具，返回中会有 tool_calls。

关键字段：

字段 含义
tool_calls[].id 工具调用 ID，第二轮返回结果时要用
tool_calls[].type 通常是 function
tool_calls[].function.name 要调用的函数名
tool_calls[].function.arguments 参数，注意通常是 JSON 字符串
finish_reason 如果是 tool_calls，说明模型还没给最终答案

特别注意：

所以后端要先解析字符串，再做参数校验。

9. 第二轮为什么要带 tool_call_id

模型第一轮可能发起多个工具调用。第二轮返回工具结果时，必须告诉模型“这个结果对应哪个调用”。

简化消息：

如果没有 tool_call_id，模型就不知道工具结果和哪个调用对应，尤其在多工具并行时会混乱。

10. Function Call 的工程边界

Function Call 看起来很智能，但工程上要管住几个边界。

风险 处理方式
模型选错工具 工具描述写清楚，后端做意图兜底
参数缺失 要求模型追问，后端校验必填参数
参数类型错误 JSON Schema + 后端校验
越权调用 后端做权限控制，不能让模型决定权限
工具执行失败 返回明确错误结果，让模型转成人话
危险操作 删除、付款、发消息等操作要二次确认

11. Function Call 和 RAG 的关系

RAG 和 Function Call 解决的问题不同。

能力 解决什么 数据来源
RAG 知识库问答 文档 Chunk、向量库
Function Call 实时数据查询和业务操作 API、数据库、业务系统

它们通常组合使用：

比如用户问：

可能需要两个能力：

  1. Function Call 查订单状态：是否签收、签收时间、商品类型。
  2. RAG 检索退货政策：是否满足退货规则。
  3. LLM 综合两部分证据生成答案。

这也是后续 Agentic RAG 的基础。

复习重点

  • Function Call 的本质是模型输出工具调用意图，不是模型真的执行代码。
  • tools 是你提供给模型的工具菜单，包含工具名、描述和参数 Schema。
  • function.description 很关键，模型主要靠它判断该不该调用工具。
  • tool_calls 表示模型要调用工具，此时通常还没有最终答案。
  • Function Call 是两轮流程：第一轮模型输出调用意图，应用执行工具；第二轮把结果交给模型生成答案。
  • arguments 通常是 JSON 字符串，后端要解析和校验。
  • 权限、安全、参数校验、危险操作确认都必须由应用代码控制。
  • RAG 负责静态知识，Function Call 负责实时数据和动作执行。

易混点整理

易混点 正确理解
Function Call 是模型调用函数 不对。模型只输出调用意图，代码执行函数
有 JSON Schema 就不用校验参数 不对。后端必须二次校验
tool_calls 就是最终答案 不对。它只是中间步骤，还要执行工具并二次请求模型
所有问题都应该走工具 不对。知识问题可以直接 RAG 或文本回答
权限可以交给模型判断 不可以。权限必须由后端系统控制
工具越多越好 不一定。工具描述、选择准确率、权限和监控都会变复杂
