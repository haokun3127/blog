学习导览

第 5 小节解决“怎么把文件解析成文本”，第 6 小节解决“怎么把长文本切成 Chunk”，第 7 小节解决“怎么给 Chunk 加来源、权限和治理信息”。第 8 小节继续往下走，解决的是：

关键词检索只能看字面是否匹配。用户问“买了一周还能退吗”，文档里写的是“自签收之日起 7 天内可申请七天无理由退货”，关键词不一定完全对得上，但语义上明显相关。Embedding 的作用，就是把文本变成向量，让系统可以按语义相似度检索。

知识速览

概念 作用 复习重点
Embedding 把文本转换成向量表示 RAG 语义检索的基础
向量 一组浮点数数组 可以理解成文本在语义空间里的坐标
语义相似度 判断两个文本含义是否接近 不是看字面完全相同，而是看含义接近
余弦相似度 衡量两个向量方向是否接近 越接近 1，通常表示越相似
Embedding 模型 负责把文本编码成向量 模型、维度、语言效果会影响召回质量
批量向量化 一批文本一起调用 Embedding API 用于提高入库效率
重新向量化 重新生成 Chunk 向量 换模型、改文档、改分块策略时要考虑

详细笔记

1. 为什么关键词检索不够用

传统搜索常见思路是关键词匹配。比如文档里出现“七天无理由退货”，用户也搜“七天无理由退货”，这类情况很好处理。

但真实问答里，用户表达非常灵活：

用户问法 文档写法 关键词检索问题
买了一周还能退吗？ 自签收之日起 7 天内可申请七天无理由退货 “买了一周”和“7 天内”不完全同词
试用期工资怎么算？ 试用期内工资为正式工资的 80% 关键词可能缺少“工资比例”
新员工多久转正？ 新员工试用期为 3 个月 “转正”和“试用期”是关联概念
发票能重新开吗？ 支持发票抬头修改 “重新开”和“抬头修改”字面不同

关键词检索的问题在于：它更关注“词有没有出现”，而不是“意思是不是接近”。Embedding 正是为了解决语义层面的匹配问题。

真实向量通常不是 3 维，而是几百维、上千维甚至更多。每一维不一定能被人类直接解释，但整体向量可以表示文本的语义特征。

可以把它想成一个语义空间：含义接近的文本，向量位置或方向更接近；含义无关的文本，向量距离更远。

3. 语义相似性怎么体现

这张图很直观：

  • “七天无理由退货”和“买了一周还能退吗”虽然字面不同，但语义接近，向量相似度很高。
  • “七天无理由退货”和“今天天气真好”语义无关，向量相似度很低。

这也是 RAG 检索比简单关键词搜索更适合问答场景的原因。用户通常不会严格按照文档原话提问，Embedding 能把“不同说法、相同含义”的内容拉近。

4. 向量维度是什么

向量维度就是这组数字的长度。

比如：

不同 Embedding 模型输出的维度可能不同。维度会影响几个方面：

维度相关问题 说明
向量库字段设计 向量数据库的字段维度必须和模型输出维度一致
存储成本 维度越高，每条向量占用空间越大
检索性能 维度越高，计算和索引成本通常越高
表达能力 维度更高不一定绝对更好，还要看模型训练质量

一个很重要的工程点：换 Embedding 模型时，要确认新旧模型的向量维度是否一致。即使维度一致，也不代表语义空间兼容，通常仍然需要重新向量化。

5. 余弦相似度

向量生成后，需要计算“两个向量有多像”。文本检索里常见指标是余弦相似度。

它关注的是两个向量方向是否接近，而不是长度是否一样。直观理解：

常见理解：

相似度 粗略含义
接近 1 很相似
接近 0 相关性弱
小于 0 方向相反，通常可以理解为很不相关

但相似度分数不是绝对真理。不同模型、不同数据集、不同问题类型下，分数分布可能不同。项目里通常要结合测试集、业务问题和召回效果来调阈值，而不是看到 0.8 就机械判断一定相关。

6. 一次向量检索的核心流程

这张图把在线检索流程拆成三步：

  1. 向量编码：用户问题先经过 Embedding 模型，变成 Query 向量。
  2. ANN 检索：向量数据库根据 Query 向量做近似最近邻检索，返回相似度最高的候选 Chunk。
  3. 元数据过滤：再根据权限、时间范围、分类标签等条件做过滤，得到最终可用结果。

这里的 ANN 是 Approximate Nearest Neighbor，近似最近邻检索。因为真实系统里可能有几十万、几百万甚至更多向量，不可能每次都暴力逐条计算相似度，所以向量数据库会通过索引结构提高检索效率。

第 8 节先理解这个三阶段即可，具体向量数据库和索引原理会在第 9 节继续展开。

7. RAG 入库时怎么使用 Embedding

离线入库链路可以这样串起来：

存入向量数据库时，通常不是只存一个向量，而是一条完整记录：

字段 内容 用途
id / chunk_id Chunk 唯一标识 更新、删除、定位
content Chunk 原文 后续拼 Prompt 给大模型
vector Embedding 向量 用于相似度检索
metadata 来源、权限、页码、版本等 用于过滤、引用和治理

所以第 7 节和第 8 节不是割裂的。元数据不参与向量化，但必须和向量一起存储。否则向量检索能召回文本，却没法做权限过滤、引用展示和问题定位。

入库向量和查询向量必须来自同一个 Embedding 模型或兼容模型。否则两边不在同一个语义空间里，相似度计算就没有意义。

10. Embedding 模型怎么选

选 Embedding 模型不是只看“谁更热门”。在 RAG 项目里更应该看这些维度：

维度 说明
中文效果 中文语义、同义表达、业务术语能否召回
向量维度 是否符合向量数据库设计和成本预算
领域适配 对代码、法律、客服、金融等特定领域是否稳定
调用成本 入库和查询都会调用，数据量大时成本明显
延迟 查询阶段会影响用户等待时间
部署方式 云端 API 省事，本地部署更可控
兼容性 模型升级后是否需要重建索引

课程里提到云端 API 和本地部署的取舍。简单记：

  • 项目初期、数据量不大、想快速验证效果：云端 API 更省事。
  • 数据量大、安全要求高、对延迟敏感：本地部署更有优势。

11. 批量向量化为什么重要

真实知识库不会只有几个 Chunk，而可能有几万、几十万甚至更多。

如果每个 Chunk 都单独调一次 API，会出现几个问题：

  • 请求次数太多，吞吐低。
  • 网络开销大。
  • 容易触发限流。
  • 失败重试难管理。
  • 整体入库时间过长。

所以入库阶段通常要做批量向量化：

批量处理要考虑几个工程点：

工程点 说明
batchSize 太小吞吐低，太大可能超过 API 限制
限流 云端 API 通常有 QPS 或 TPM 限制
重试 网络失败、服务抖动要能自动重试
幂等 同一批重复执行不能产生脏数据
进度记录 大文档入库失败后能从断点继续
错误隔离 某个 Chunk 异常不能拖垮整批任务

12. 什么时候需要重新向量化

向量化不是一次做完永久有效。下面几种情况需要特别注意。

12.1 换 Embedding 模型

不同模型生成的向量通常不兼容。

即使两个模型都输出 1024 维向量，它们的语义空间也可能完全不同。模型 A 的向量不能随便和模型 B 的向量一起算相似度。

结论：

12.2 文档内容更新

如果原文从“7 天内可退”改成“15 天内可退”，对应 Chunk 文本已经变了，旧向量也不再准确。

常见处理：

  1. 根据 doc_id 找到旧 Chunk。
  2. 删除旧向量或标记失效。
  3. 重新解析、分块、补元数据。
  4. 重新生成向量并入库。

这也再次说明第 7 节的 doc_id 很重要。没有文档标识，就很难批量找到某份文档对应的所有旧向量。

12.3 分块策略调整

如果 chunkSize、overlap 或分块算法改变了，Chunk 内容会变化，向量自然也要重建。

比如原来 500 字一块，后来改成 300 字一块，同一份文档会切出完全不同的 Chunk 集合。旧向量已经对应不上新的 Chunk。

12.4 模型版本升级

同一个模型不同版本也可能不兼容。实际项目里最好在元数据里记录：

这样后续排查召回问题、灰度升级模型、重建索引时都更清楚。

13. Embedding 的边界

Embedding 很重要，但它不是万能的。

它能解决：

  • 用户问法和文档说法不同。
  • 同义表达召回。
  • 语义相近内容匹配。
  • 大规模文本的候选召回。

它不能单独解决：

  • 权限过滤。
  • 版本时效判断。
  • 多条结果冲突。
  • 复杂推理。
  • 答案生成和表达。
  • 引用来源展示。

所以 RAG 通常不是“Embedding + 向量库”就结束了。后面还需要元数据过滤、Rerank、Prompt 拼装、LLM 生成、引用溯源和质量评估。

学习重点

  • Embedding 是把文本变成向量，让系统能做语义相似检索。
  • 关键词检索看字面匹配，Embedding 检索看语义接近。
  • 入库阶段和查询阶段要使用同一个或兼容的 Embedding 模型。
  • 向量数据库里通常要一起保存 content、vector 和 metadata。
  • 余弦相似度常用来衡量向量方向是否接近，但阈值需要结合业务测试调。
  • 换模型、改文档、改分块策略，都可能需要重新向量化。

易混点整理

易混点 正确理解
Embedding 是生成答案吗？ 不是。Embedding 只负责把文本编码成向量，答案仍由 LLM 生成
相似度高就一定正确吗？ 不一定。还需要权限、时间、业务标签、Rerank 和模型判断
元数据会参与向量化吗？ 通常不参与，但要和向量一起存储，用于过滤和展示
换 Embedding 模型能直接复用旧向量吗？ 通常不行，需要重新向量化
向量维度越高越好吗？ 不一定。要看模型质量、成本、存储和检索性能

和前后章节的关系

到第 8 节为止，RAG 的数据准备链路已经基本闭环。下一节的重点会转到“向量生成之后怎么存、怎么索引、怎么快速查”。

一句话总结

Embedding 让 RAG 从“按关键词找文本”升级为“按语义找知识片段”，是向量检索能够成立的前提。
