详细笔记

1. 为什么不能把整篇文档直接塞进向量库

很多人第一次接触 RAG 时，容易有个很自然的想法：

问题在于，整篇文档通常太长、信息太杂，拿它作为一个检索单位会带来几个明显问题。

第一，颗粒度太粗

如果一整篇制度文档、产品说明书或者操作手册只生成一个向量，那么用户问的其实只是其中一小段内容，但检索命中的却是整篇文档。这样虽然“文档相关”，但真正有用的信息并不集中，后续交给模型时上下文会很松散。

第二，噪声太大

长文档里通常包含很多和当前问题无关的内容。把整篇都作为一个检索单元，会让无关信息一起进入召回结果，影响检索精度，也会让模型在生成时更难抓住重点。

第三，生成时上下文利用率低

即使模型能接住很长上下文，也不代表越长越好。大量无关文本会消耗 Token，让真正关键的片段被稀释。

你给的这张图正好把这条链路画得很清楚：

这张图的重点不是“分块只是中间一步”，而是告诉我们：

  • 分块以前，系统面对的是整段原始文本
  • 分块以后，系统面对的是一批可检索的小片段
  • 后面的 Embedding 和向量库，处理的都是 Chunk，而不是原文全文

所以 Chunk 是整个 RAG 知识库的基本颗粒度单位。颗粒度设得是否合理，会直接影响后续召回质量。

3. 分块到底在解决什么问题

分块不是为了“把长文切开看起来舒服”，而是在解决以下几个核心问题：

1. 让检索结果更精确

用户问的是一个局部问题，系统最好返回局部相关片段，而不是整篇资料。

2. 让向量表达更聚焦

一个 Chunk 内容越集中，它的向量语义通常越明确；如果一个 Chunk 混了太多主题，向量表达就会模糊。

3. 让模型拿到更可控的上下文

RAG 最终是把若干个召回 Chunk 拼进 Prompt。Chunk 如果过大，就会快速占满上下文；如果过小，就得拼很多块，反而容易碎。

4. 降低系统成本

分块会影响：

  • 向量化次数
  • 向量库存储量
  • 检索召回数量
  • Rerank 输入数量
  • 最终 Prompt 长度

所以分块不是纯算法问题，它同时还是成本问题和工程问题。

4. chunkSize 为什么很关键

chunkSize 可以理解成“每个 Chunk 大概多长”。

这个长度没有一个固定标准，因为它会受到很多因素影响：

  • 文档类型
  • 业务问法
  • 模型上下文大小
  • Embedding 模型特性
  • 后续 Rerank 和 Prompt 拼装策略

但它的取舍逻辑很清楚。

Chunk 太大时

会有这些问题：

  • 一个 Chunk 里塞了太多主题
  • 检索命中了块，但真正答案只占其中很小一部分
  • 向量语义不够聚焦
  • 后面 Prompt 太长

Chunk 太小时

会有这些问题：

  • 一句话被切得太碎
  • 上下文关系丢失
  • 条款前后条件断开
  • 召回结果很多但信息不完整

5. overlap 为什么存在

如果我们硬切分，比如每 500 个字符切一刀，就很容易把一句话、一段条款或一个逻辑单元从中间切断。

这时 overlap 就派上用场了。

overlap 指的是：相邻两个 Chunk 之间，保留一部分重叠内容。

例如：

这里重叠的就是 50 个字符。

它的作用是：

  • 减少边界切断带来的信息丢失
  • 让前后片段之间保留一点上下文连续性
  • 提高边界位置内容被正确召回的概率

但 overlap 也不是越大越好。

如果 overlap 太大，会带来：

  • 向量数量增加
  • 存储成本上升
  • 重复内容变多
  • 检索时更容易召回多个高度重复的块

6. 主流分块策略有哪些

固定大小分块

最简单的方式，就是按固定字符数或 Token 数切分。

优点：

  • 实现简单
  • 处理速度快
  • 适合快速验证

缺点：

  • 很容易从中间切断语义
  • 不关心段落和句子结构

适合：

  • 原型阶段
  • 文本结构简单
  • 先跑通链路

递归分块

递归分块会优先按照更自然的结构边界切，比如：

也就是说，能按段落切就先按段落切；如果段落太长，再往下按句子切；句子还太长，再继续细分。

优点：

  • 更尊重文档原始结构
  • 语义完整性更好

缺点：

  • 规则设计会更复杂

适合：

  • 普通文档
  • 制度类、说明类、正文型资料

语义分块

语义分块不是只看字符数，而是看内容语义是否自然结束。

优点：

  • 语义边界更自然
  • 检索质量潜力更高

缺点：

  • 实现成本高
  • 计算成本高
  • 对系统工程复杂度要求更高

适合：

  • 高价值知识库
  • 对回答精度要求高的场景

混合分块

企业级系统更常见的是混合策略：

  • 先利用标题、页码、段落等结构信息
  • 再配合长度上限
  • 再补 overlap
  • 必要时按业务类型定制规则

这类方案更接近真实生产环境，因为企业资料通常不会只有一种格式和一种内容结构。

7. 不同文档类型，分块策略也不一样

这一点特别重要。

政策制度类文档

更适合保留条款、标题、编号的完整性。因为用户常常会问：

  • 某一条规定是什么
  • 某个条件成立时怎么处理
  • 某项制度适用于谁

如果把条款从中间切断，会严重影响回答质量。

FAQ 或问答类文档

更适合一问一答作为一个 Chunk，因为问答本身就是天然语义单元。

产品说明书或操作手册

通常适合按步骤段落、功能模块、标题层级切分。

表格型资料

如果强行按字符切，可能把表头和表体拆开，造成后续理解错误。

8. 分块不是小细节，而是召回质量的核心变量

很多人会把分块看成入库前随手做的一步，但它实际上直接决定召回效果。

因为用户问问题时，向量库返回的不是文档，而是 Chunk。

如果 Chunk 切得不合理，就会出现：

  • 明明文档里有答案，却召回不到
  • 召回到了很多块，但每块都不完整
  • 召回的块语义混杂，模型不好组织回答

所以从“检索质量”这个角度看，Chunk 其实是一个非常核心的调参点。

很多时候，RAG 回答不准，问题不一定在模型，也可能在：

  • 解析质量差
  • 分块策略不合理
  • overlap 设置有问题
  • 召回后上下文过碎或过粗

学习重点

  • Chunk 是 RAG 检索和回答的基本单位，不是整篇文档。
  • 分块的目标是在语义完整性、检索精度、存储成本、向量化成本之间做平衡。
  • chunkSize 太大太小都不行，本质上要让 Chunk “既完整又聚焦”。
  • overlap 用来缓解边界切断问题，但会引入重复和成本。
  • 固定分块、递归分块、语义分块各有适用场景。
  • 企业级分块通常采用混合策略，而不是只按固定长度硬切。

易混点整理

1. 分块不是为了排版好看

分块的核心目的是提升检索质量和上下文可用性，不是把长文切开看着舒服。

2. chunkSize 没有通用标准答案

它必须结合文档类型、业务问法、模型上下文和系统成本一起调整。

3. overlap 不是越大越好

它能减少断裂，但也会增加重复、存储和检索成本。

4. 固定分块简单，但不一定适合生产

快速验证时很好用，但真实业务里通常还要结合段落、标题、条款、问答结构做增强。

5. 分块策略会直接影响召回质量

RAG 回答不准，不一定是模型问题，很多时候就是 Chunk 切得不合理。

一句话总结

第 6 小节讲清楚了 RAG 入库链路里最容易被低估、但又最关键的一步：文档解析后的文本不能直接整篇入库，而要切成“语义尽量完整、长度尽量合适、便于检索召回”的 Chunk，这一步的质量会直接影响后面的向量化、检索和最终回答效果。
