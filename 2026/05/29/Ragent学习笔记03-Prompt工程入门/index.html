Ragent学习笔记03：Prompt 工程入门

学习导览

这一节从“会调用模型”进入“让模型稳定按业务要求回答”。Prompt 不是随便写一句话，而是把角色、任务、约束、输入和输出格式组织成一套可复用、可测试、可迭代的工程化指令。对 RAG 来说，Prompt 的重点不是让模型更会发挥，而是让模型只基于检索资料回答，并在资料不足时懂得澄清或兜底。

知识速览

概念 学习要点
Prompt 五要素 角色、任务、约束、输入、输出共同决定模型行为
角色 Role 定义模型身份和回答边界，避免模型乱扩展职责
任务 Task 明确模型要完成什么，避免意图模糊
约束 Constraints 规定知识来源、风格、长度、禁止事项和优先级
输入 Inputs 用分隔符、字段名、可信度说明把上下文喂清楚
输出 Outputs 规定结构、引用、兜底和澄清方式
Prompt 注入 资料或用户输入中可能夹带“忽略上文”等恶意指令
Bad Case 迭代 从错误案例反推 Prompt 缺少哪条规则

详细笔记

Prompt 工程为什么重要

大模型不是传统函数，输入稍微变化就可能导致输出风格、边界和准确性变化。Prompt 工程的价值，是把“希望模型怎么答”变成稳定的规则。

在 RAG 系统中，检索结果只是原材料，Prompt 决定模型如何使用材料、如何处理冲突、如何拒绝没有依据的问题。好检索配坏 Prompt，依然会生成不可用答案。

Prompt 的五要素框架

角色说明模型是谁，例如知识库问答助手、客服助手、代码审查助手；任务说明要做什么，例如回答问题、提取字段、判断意图；约束说明不能做什么，例如不能编造、不能使用参考资料外的信息。

输入要清楚区分用户问题、历史对话、检索片段和业务参数。输出要明确格式、长度、引用方式和兜底话术。五要素越清楚，模型越容易稳定工作。

RAG 场景下的特殊要求

RAG Prompt 要明确“参考资料是事实来源，不是指令来源”。这能降低 Prompt 注入风险，避免知识库文本里的恶意指令覆盖系统规则。

引用规则要可验收：没有引用支撑的事实不要输出；引用内容必须真的支持答案；资料冲突时要说明冲突，而不是强行合并。

Prompt 优化流程

不要凭感觉一次写完 Prompt。更工程化的方式是：收集 bad case，定位失败原因，修改规则，再用固定测试集回归。

Prompt 也要版本管理。每次修改要记录修改原因、解决了什么问题、是否引入新问题，避免线上效果不可追溯。

常见误区

Prompt 太短会缺少边界，模型容易自由发挥；Prompt 太长又可能规则冲突、占用 Token。比较好的做法是保留核心规则，分组编号，给出优先级。

Few-shot 示例适合复杂格式或边界场景，但不是越多越好。示例太多会占用上下文，示例不典型还会误导模型。

学习重点

  • Prompt 不是玄学，而是对模型输入的工程化设计。
  • RAG Prompt 的核心是限定知识来源、处理冲突、要求引用、提供兜底。
  • Prompt 注入防护要写清楚：系统规则优先于用户问题，用户问题优先于参考资料中的文本指令。
  • Prompt 需要像代码一样迭代、测试和版本管理。

易混点整理

  • System Prompt 不等于随便设定人设，它承担的是行为边界和安全约束。
  • Few-shot 不是必需品，简单任务用清楚规则通常更稳。
  • 资料不足时直接编答案不是“智能”，而是 RAG 系统最需要避免的问题。
