Ragent学习笔记04：什么是 RAG

发表于2026-05-29|更新于2026-06-01|Ragent

学习导览

这一节正式进入 RAG。前面已经知道大模型可以根据 Prompt 生成答案，但真实业务里不能只靠模型自己的“记忆”。企业内部文档、产品手册、规章制度、接口文档、订单数据，这些内容大模型训练时不一定见过，即使见过也可能已经过期。

RAG 的核心思路是：不要急着让模型直接回答，先从外部知识库里检索相关资料，再把资料和用户问题一起交给模型生成答案。

所以 RAG 可以理解为一种“检索 + 生成”的组合方案。它不是重新训练模型，也不是把所有文档一股脑塞给模型，而是在回答问题之前，给模型临时补充一份和问题相关的参考资料。

知识速览

概念 学习要点
RAG Retrieval-Augmented Generation，检索增强生成
Retrieval 检索，从知识库中找出和用户问题最相关的资料片段
Augmented 增强，把检索到的资料作为上下文补充给大模型
Generation 生成，大模型基于问题和资料组织自然语言答案
知识库 存放企业文档、FAQ、手册、政策、技术资料等内容
Chunk 文档切分后的文本块，是检索和喂给模型的基本单位
Embedding 把文本转换成向量，让系统可以按语义相似度检索
向量数据库 存储文本向量并支持相似度检索的数据库
Prompt 拼装 把用户问题、检索结果、回答规则组合成最终模型输入
引用溯源 标明答案来自哪个资料片段，方便验证答案依据
幻觉缓解 通过资料约束减少模型编造，但不能完全自动消除幻觉
企业级 RAG 还要处理权限、版本、限流、监控、记忆、工具调用和反馈优化

详细笔记

1. RAG 是什么

RAG 的全称是 Retrieval-Augmented Generation，中文一般叫“检索增强生成”。

拆开看更好理解：

部分 含义 在系统里做什么
Retrieval 检索 先从知识库中找资料
Augmented 增强 把资料补充到模型上下文里
Generation 生成 让大模型基于资料生成答案

一句话概括：

它的重点不是让模型“记住”更多知识，而是让模型在回答前“看到”相关资料。

这张图很好地说明了 RAG 的作用：普通 LLM 遇到训练后发生的新事件时，可能会因为没有相关知识而拒答或答错；加入 RAG 后，系统会先从文档中检索相关片段，再把“问题 + 相关文档”组合成 Prompt 交给大模型生成答案。这样模型不是凭空猜，而是带着资料回答。

这个类比很重要。RAG 并没有改变模型参数，它只是改变了模型回答问题时看到的上下文。

2. 为什么需要 RAG

大模型本身很强，但直接用于企业问答会遇到几个明显问题。

问题 说明 RAG 的解决思路
知识过期 模型训练完成后，参数里的知识不会自动更新 把最新资料放进知识库，查询时实时检索
私有知识缺失 模型不知道公司内部制度、产品文档、项目代码 把企业私有文档入库，回答时检索出来
幻觉问题 模型可能用听起来合理的话编答案 用参考资料约束模型，只允许基于资料回答
不可追溯 直接回答时很难知道依据来自哪里 答案标注引用，能追溯到原文
训练成本高 为了新知识重新训练或微调模型成本很高 不改模型参数，只更新知识库
上下文有限 不能把所有资料都塞进 Prompt 只检索和问题相关的片段

这也是 RAG 在企业应用里很常见的原因：它比重新训练模型轻量，又能把企业知识接进大模型。

3. RAG 和微调有什么区别

RAG 和微调都能让模型更适合某个业务场景，但解决的问题不一样。

对比项 RAG 微调
核心方式 检索外部知识，作为上下文喂给模型 用数据继续训练模型参数
适合内容 经常更新的知识、企业文档、FAQ、政策 固定能力、输出风格、特定任务模式
更新成本 更新知识库即可 通常需要重新训练或重新发布模型
可追溯性 强，可以引用资料来源 弱，知识融进参数后不容易追溯
典型场景 知识库问答、文档助手、客服问答 分类、格式化输出、领域风格适配

我的理解是：

如果知识经常变化，优先考虑 RAG。如果希望模型学会某种稳定的输出格式、专业语气或任务模式，可以考虑微调。

5. 离线入库链路

离线入库解决的问题是：怎么把文档变成可检索的数据。

5.1 文档解析

企业文档不一定都是干净文本，可能是：

  • PDF
  • Word
  • PPT
  • Excel
  • Markdown
  • HTML 网页
  • 数据库记录
  • 扫描件图片

解析阶段要把这些格式转换成文本。后面第 5 小节会学 Apache Tika，它就是用来解析多种文档格式的工具。

文档解析质量很关键。如果解析出来的文本乱码、缺表格、顺序错乱，后面检索和生成都会受影响。

5.2 文本清洗

解析后通常还要清洗，比如：

  • 去掉页眉页脚
  • 去掉无意义空行
  • 处理重复内容
  • 修复乱码
  • 保留标题层级
  • 保留表格结构

清洗不是为了好看，而是为了让后续分块和检索更稳定。

5.3 文档分块 Chunk

大模型和向量检索都不适合直接处理整篇长文档，所以要把文档切成较小的片段，这个片段就是 Chunk。

分块的难点是平衡：

分块方式 问题
切太大 检索不精准，一个 chunk 里可能只有一句有用
切太小 上下文丢失，检索到片段后看不懂
固定长度切 简单，但可能把语义完整的一段切断
按标题/段落切 语义更自然，但实现更复杂

所以 Chunk 策略会影响召回质量。后面第 6 小节会专门学数据分块。

5.4 元数据 Metadata

元数据是描述 chunk 的附加信息，比如：

元数据 作用
文档 ID 知道 chunk 来自哪个文档
标题 用于展示和溯源
章节 保留文档结构
更新时间 处理新旧版本冲突
权限范围 控制哪些用户可以检索
来源类型 区分官方文档、FAQ、工单、网页

元数据不是可有可无的装饰，它会参与过滤、排序、权限控制和引用展示。后面第 7 小节会展开。

Embedding 是把文本转换成向量。向量可以理解成文本的语义坐标。

6.3 TopK 和重排

检索通常会返回 TopK 个结果，比如前 5 条、前 10 条。

TopK 不是越大越好：

TopK 影响
太小 可能漏掉关键资料
太大 噪声变多，Prompt 更长，模型更容易被干扰

很多系统会先召回一批候选 chunk，再用 Rerank 模型重新排序。Rerank 的作用是用更精细的模型判断哪些片段最适合回答当前问题。

6.5 生成答案和引用溯源

生成阶段的目标不是让模型随意发挥，而是让模型基于资料组织答案。

比较好的答案应该满足：

  • 直接回答用户问题
  • 关键事实有引用
  • 不补充资料里没有的信息
  • 资料不足时不编造
  • 如果资料冲突，说明冲突或按规则处理

引用溯源是 RAG 的重要价值。用户看到答案后，可以点回原文确认依据。

7. RAG 能缓解幻觉，但不能自动消除幻觉

RAG 可以降低幻觉，因为模型有了外部资料作为依据。但它不能保证完全没有幻觉。

原因有几个：

幻觉来源 说明
检索错了 召回的资料本身不相关
检索漏了 关键资料没有被召回
资料过期 知识库里有旧版本文档
Prompt 约束弱 模型仍然使用预训练知识补充
引用不严格 模型可能空挂引用
用户问题模糊 系统没问清楚就强行回答

所以 RAG 还需要配套机制：

  • 更好的文档入库质量
  • 更合理的检索策略
  • Rerank 重排序
  • 强约束 Prompt
  • 引用校验
  • 答案评估
  • 用户反馈闭环

8. RAG 适合哪些场景

RAG 适合“答案依赖外部资料，而且资料需要更新或追溯”的场景。

场景 适合原因
企业知识库助手 公司制度、流程、规范都在内部文档里
智能客服 产品文档、FAQ、售后规则可以入库
法律助手 回答需要依据法条、判例和合同条款
金融研报问答 需要基于财报、公告、研报回答
医疗文献助手 需要依据指南、文献和药品说明
代码/技术文档助手 可以检索代码仓库、API 文档、项目说明

RAG 不适合所有问题。比如：

  • 纯闲聊不需要 RAG
  • 简单通识问题不一定需要 RAG
  • 需要实时操作业务系统的问题，只靠 RAG 不够

9. 企业级 RAG 的难点

跑通一个 demo 不难，难的是把它做成稳定、可用、可维护的系统。

9.1 数据入库难

文档来源复杂，格式不统一。PDF、Word、网页、表格、扫描件都可能出现。

难点包括：

  • 解析不干净
  • 表格丢结构
  • 标题层级丢失
  • 文档版本混乱
  • Chunk 切分不合理
  • 元数据缺失

数据入库质量决定 RAG 的上限。前面处理不好，后面模型再强也很难答好。

9.2 问题重写难

用户不会按知识库里的标准术语提问。

常见情况：

  • 口语化表达
  • 省略上下文
  • 多轮追问
  • 一个问题包含多个意图
  • 复杂问题需要拆成多个子问题

问题重写的本质是弥补“用户表达”和“检索表达”之间的差距。

9.3 意图识别和路由难

不是所有问题都应该走同一条链路。

系统要判断：

  • 是闲聊还是知识问答
  • 是查知识库还是调用工具
  • 应该查哪个知识库
  • 是否需要多知识库并行检索
  • 是否涉及敏感信息或越权访问

意图识别错了，后面的检索和生成都会跟着错。

9.4 检索难

检索不是简单的“相似度最高就最好”。

问题包括：

  • 向量检索不擅长精确编号
  • 关键词检索不擅长语义相似
  • TopK 太小会漏召回
  • TopK 太大会引入噪声
  • 旧文档和新文档可能冲突
  • 用户可能没有权限看某些资料
  • 召回结果需要重排和过滤

这也是为什么后面会继续学向量数据库、检索策略和召回优化。

9.5 多轮记忆难

多轮对话会带来上下文管理问题。

如果把所有历史都塞给模型，Token 成本高，而且容易干扰答案。如果只带最近几轮，又可能丢掉关键信息。

常见策略包括：

  • 只保留最近 N 轮
  • 对历史对话做摘要
  • 把关键记忆持久化
  • 根据当前问题检索相关历史

这和后面的多轮对话记忆设计会接上。

9.6 工程化难

生产环境还要考虑很多非算法问题：

  • 权限控制
  • 请求限流
  • 模型负载均衡
  • 第三方 API 降级
  • 停止生成
  • 日志和链路追踪
  • 用户反馈收集
  • 效果监控
  • 知识库版本管理

企业级 RAG 不是一个向量数据库加一个大模型，而是一套完整的问答工程系统。

10. RAG 在 Ragent 项目中的位置

学完这一节后，对 Ragent 的主线会更清楚。

Ragent 不是简单调用大模型 API，而是围绕企业级知识问答，把 RAG 相关能力串起来：

后面的章节基本就是把这条链路逐个拆开：

后续主题 对应 RAG 环节
Apache Tika 文档解析
Chunk 策略 文档分块
元数据管理 过滤、权限、溯源
Embedding 文本向量化
向量数据库 向量存储和检索
向量检索策略 召回优化
生成策略与幻觉抑制 让答案更可靠
Function Call / MCP 查询业务系统或调用外部工具
多轮记忆 处理连续对话
查询重写 提升检索效果
意图识别与路由 决定查哪个库、走哪条链路
SSE 流式响应 把生成结果实时返回前端

11. 一句话复述 RAG

这一节可以用这句话收束：

更面向项目的说法是：

学习重点

  • RAG 的全称是 Retrieval-Augmented Generation，核心是“先检索，再生成”。
  • RAG 不会修改模型参数，它是在回答时给模型补充外部资料。
  • RAG 适合企业知识库、客服、专业领域问答、技术文档助手等场景。
  • RAG 的离线链路是：文档解析、清洗、分块、Embedding、入向量库。
  • RAG 的在线链路是：问题理解、检索召回、过滤重排、Prompt 拼装、模型生成、引用溯源。
  • RAG 可以缓解幻觉，但不能自动消除幻觉，仍然需要检索质量、Prompt 约束和评估机制。
  • 企业级 RAG 的难点不止在模型，还包括数据入库、检索策略、权限、记忆、限流、监控和反馈闭环。
  • Ragent 后续章节就是围绕这条 RAG 链路逐步展开。

易混点整理

  • RAG 不是训练模型：它不改变模型参数，只是在推理时补充资料。
  • 向量数据库不等于 RAG：向量数据库只是检索组件，完整 RAG 还包括文档处理、Prompt、生成、溯源和工程化。
  • Embedding 不是答案：Embedding 只是为了检索相似文本，真正的答案还要靠模型生成。
  • Chunk 不是随便切：分块大小和边界会直接影响检索质量。
  • 检索到资料不代表答案可靠：还要看资料是否相关、是否过期、是否有权限、是否被正确引用。
  • RAG 不能完全消灭幻觉：检索错、Prompt 弱、资料冲突时仍然可能出问题。
  • RAG 和工具调用不是一回事：RAG 适合查文档，工具调用适合查实时业务数据或执行操作。
  • 企业级 RAG 不是 demo：真正上线要考虑权限、限流、负载、监控、反馈和版本管理。
