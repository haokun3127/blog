Ragent学习笔记02：调用大模型 API

发表于2026-05-28|更新于2026-06-01|Ragent

  • 《AI大模型Ragent项目》第2小节：调用大模型API
  • 本地文档路径：C:\Users\ASUS\Desktop\项目代码\ragent\文档\《AI大模型Ragent项目》第2小节：调用大模型API.html

学习导览

这一节开始从“认识大模型”进入“真正调用大模型”。核心目标是：用 Java 程序通过 HTTP 请求调用大模型 API，理解请求 JSON 怎么写、响应 JSON 怎么解析，以及非流式和流式两种调用方式分别适合什么场景。

本节最重要的结论是：大模型 API 本质上也是一个 HTTP 接口。只要掌握 OpenAI 兼容协议，就可以用同一套请求格式调用 SiliconFlow、DeepSeek、Qwen 等不同平台或模型。

知识速览

概念 学习要点
OpenAI 兼容协议 大模型 API 的事实标准，换平台通常只改 baseURL 和 API Key
model 指定本次调用使用哪个模型
messages 对话消息数组，是 Chat API 的核心输入
system 定义模型身份、规则和回答边界
user 用户输入的问题
assistant 历史模型回答，用于构建多轮对话上下文
temperature 控制随机性，RAG 问答通常设置较低
max_tokens 控制最大输出长度，太小会导致回答被截断
stream 控制是否流式返回，false 非流式，true 流式
SSE Server-Sent Events，服务端持续向客户端推送数据

详细笔记

OpenAI 接口协议是大模型 API 的“普通话”

虽然课程里使用的是 SiliconFlow 平台上的 Qwen 模型，但先学习 OpenAI 接口协议很有必要。原因是 OpenAI 的 Chat Completions API 已经成为大模型 API 的事实标准，很多平台都会提供 OpenAI 兼容接口。

这意味着后续切换模型或平台时，通常不需要重写业务逻辑，只需要改：

  • baseURL
  • API Key
  • model

例如从 SiliconFlow 切到 DeepSeek，本质上就是换 API 地址和密钥，请求体结构仍然是 model + messages + 参数 这一套。

Chat API 请求体结构

调用大模型 API，本质上是发一个 HTTP POST 请求，请求体是 JSON。

典型结构如下：

可以把这个请求体理解成一次“带规则的对话调用”：

  • model 决定找哪个模型回答。
  • messages 决定模型能看到什么上下文。
  • temperature 决定回答稳定还是发散。
  • max_tokens 限制模型最多生成多少内容。
  • stream 决定一次性返回还是边生成边返回。

messages 角色机制

messages 是 Chat API 最核心的字段，它是一个数组，每条消息都有 role 和 content。

角色 作用 RAG 中的意义
system 定义模型身份、任务规则和回答边界 告诉模型必须根据参考资料回答，资料没有就不要编
user 用户当前输入的问题 表示用户真正想问什么
assistant 模型之前的回答 构建多轮对话上下文

大模型 API 本身没有长期记忆。所谓多轮对话，其实是应用系统每次请求时，把前面的 user 和 assistant 历史消息重新带上，让模型“临时看到”之前聊过什么。

这也带来一个重要问题：对话越长，messages 越长，消耗的 Token 越多。所以后续 Ragent 做多轮对话记忆时，需要考虑历史消息保留、摘要压缩和 Token 成本。

system 消息非常关键

system 消息可以从根本上改变模型的行为模式。比如同样问“Java 和 Python 哪个好”，不同的 system 会让模型变成技术顾问、Java 粉丝，或者用儿童语气回答。

常用请求参数

参数 含义 RAG 场景建议
temperature 控制回答随机性 0 到 0.3，保证稳定准确
max_tokens 最大输出 Token 数 根据回答长度设置，常见 512 到 2048
top_p 另一种随机性控制方式 通常和 temperature 二选一
stream 是否启用流式返回 用户聊天用 true，后台任务用 false

如果经常出现回答没说完就停了，要检查响应里的 finish_reason 是否为 length。如果是，通常说明 max_tokens 设置太小。

非流式调用

非流式调用就是 stream=false。模型会等完整答案生成完毕后，一次性返回一个完整 JSON。

适合场景：

  • 后台批量处理
  • 文档摘要
  • Embedding、Rerank 这类不需要实时展示的任务
  • 开发调试阶段

Java 实现思路：

  1. 用 Gson 构建请求 JSON。
  2. 用 OkHttp 创建 HTTP 客户端。
  3. 添加 Authorization: Bearer API_KEY 请求头。
  4. 发送 POST 请求。
  5. 解析响应里的 choices[0].message.content。
  6. 读取 usage 统计 Token 消耗。

非流式调用的优点是简单、好调试，拿到的是完整 JSON；缺点是用户必须等模型生成完，页面才会看到结果。

非流式响应格式

非流式响应中，模型回答在 choices[0].message.content。

核心字段：

字段 含义
id 本次请求 ID，可用于日志追踪
choices 模型输出数组，通常取第一个
choices[0].message.content 模型完整回答
choices[0].finish_reason 模型停止原因
usage.prompt_tokens 输入消耗 Token
usage.completion_tokens 输出消耗 Token
usage.total_tokens 总 Token 消耗

usage 对成本监控很重要，因为 API 通常按 Token 计费。

finish_reason 常见值：

值 含义 处理方式
stop 正常结束 不需要处理
length 达到 max_tokens 上限 调大 max_tokens 或缩短 Prompt

流式调用

流式调用就是 stream=true。模型不是等所有内容生成完才返回，而是每生成一小段就推送给客户端。

用户看到的效果就是“打字机效果”：答案一段一段出现，而不是等待很久后突然出现完整回答。

注意：流式调用不一定让模型总生成时间变短，但会显著降低用户的等待感，因为首个 Token 可以更早展示。

这张图可以帮助记住流式调用的链路：Java Client 发起 POST /chat/completions 请求，请求体里打开 stream=true；SiliconFlow API 再向 LLM 发起推理请求；模型每生成一小段 Token，就通过 SSE Event 往客户端返回；最后以 data: [DONE] 作为流式结束信号。

适合场景：

  • 面向用户的聊天窗口
  • RAG 问答页面
  • 长文本回答
  • 深度思考或复杂推理场景

SSE 协议

流式调用通常基于 SSE（Server-Sent Events）。

普通 HTTP 是“一问一答”：客户端发请求，服务端返回完整响应，然后连接关闭。

SSE 是“服务端持续推送”：客户端发请求后，服务端保持连接打开，不断返回以 data: 开头的数据块，直到最后返回：

[DONE] 表示本次流式输出结束。

流式响应解析时要注意：

  • 每行通常以 data: 开头。
  • 空行要跳过。
  • data: [DONE] 不是 JSON，不能拿去 JSON 解析。
  • 第一个数据块可能只有 role，没有正文内容。
  • 结束前可能出现 finish_reason=stop。

流式 Java 代码关键点

和非流式相比，流式调用有几个关键区别：

关键点 说明
stream=true 请求体中必须开启流式
逐行读取 不能用 response.body().string() 一次性读取
解析 delta 增量内容在 choices[0].delta.content
处理 [DONE] 收到后结束读取
实时输出 控制台用 System.out.print 模拟打字效果
超时时间更长 流式连接保持时间更久，readTimeout 要设置长一些

非流式和流式怎么选

对比维度 非流式 stream=false 流式 stream=true
返回方式 完整生成后一次性返回 边生成边推送
首字延迟 高 低
用户体验 等待感强 打字机效果，更顺滑
实现复杂度 简单 需要解析 SSE
响应格式 一个完整 JSON 多个 data: 数据块
Token 用量 通常直接有 usage 部分平台流式不返回 usage
适用场景 后台任务、调试、批处理 聊天页面、RAG 问答、长回答

实用建议：开发阶段先用非流式，因为完整 JSON 更方便调试；功能跑通后，再把面向用户的问答部分改成流式。

本节和 RAG 的关系

后续 RAG 系统中会反复使用这一套 API 调用套路：

  • Chat API：根据用户问题和检索资料生成回答。
  • Embedding API：把文本转成向量。
  • Reranker API：对召回结果重新排序。

虽然这三类 API 的具体请求字段不同，但核心思路一致：构建 JSON 请求体，发 HTTP POST，解析 JSON 响应。

学习重点

  • OpenAI Chat Completions API 是大模型 API 的事实标准。
  • messages 是 Chat API 的核心，system、user、assistant 分别承担不同角色。
  • 大模型本身没有长期记忆，多轮对话是应用系统把历史消息重新带给模型。
  • system 消息在 RAG 中用于约束模型根据资料回答，避免编造。
  • 非流式调用简单，适合后台处理和开发调试。
  • 流式调用基于 SSE，适合用户聊天和 RAG 问答页面。
  • 非流式读取 message.content，流式读取 delta.content。
  • Java 中用 OkHttp + Gson 就能完成基础模型调用。

易混点整理

  • OpenAI 兼容协议不是只能调用 OpenAI，它是一套很多厂商都兼容的接口格式。
  • messages 不是只放当前问题，多轮对话需要把历史消息也放进去。
  • assistant 角色不是你手写答案，而是用于携带模型之前的回答。
  • stream=true 不代表模型总耗时一定变短，只是让用户更早看到部分输出。
  • [DONE] 是 SSE 结束标记，不是 JSON。
  • finish_reason=length 说明回答可能被截断，不是模型正常讲完。
  • API Key 相当于密码，不能提交到 Git，也不能写进公开代码。
