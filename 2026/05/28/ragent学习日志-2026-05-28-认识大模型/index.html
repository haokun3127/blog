ragent复习笔记：认识大模型

复习导览

这一节主要是在正式写 RAG 代码之前，先建立大模型的基础认知：大模型是什么、为什么比传统规则程序更适合自然语言场景、常见模型有哪些、实际项目中应该调用哪类模型，以及为什么开发者最终需要通过 API 把大模型接入自己的系统。

文章从“客服系统接入 AI”的业务场景切入：网页端的大模型很好用，但真正做项目时，Java 系统不能依赖人工在网页里复制粘贴问题，而是要通过 API 把大模型变成系统里的一个能力组件。

一页速览

概念 复习抓手
大模型 不靠人工写死规则，而是从海量文本中学习语言规律
训练阶段 预训练得到 Base Model，指令微调和对齐训练得到更适合对话的 Chat Model
参数量 7B、14B、72B 表示参数规模，能力更强通常成本也更高
Token 大模型处理文本的基本单位，影响计费和上下文长度
上下文窗口 模型一次能看到的总文本量，RAG 需要检索后只放关键片段
Temperature 控制回答随机性，RAG 场景一般设置较低
MoE 总参数很多，但每次只激活部分专家参数，降低推理成本
Chat 模型 RAG 实际调用的模型类型，能理解指令并基于上下文回答
量化 用更低精度压缩权重，降低本地部署的显存和存储成本
API 调用 把大模型接入 Java/RAG 系统的核心方式

详细笔记

1. 大模型和传统程序的区别

传统业务程序通常依赖开发者提前写好规则，例如用 if-else 或关键词判断用户是不是想退货、查物流、咨询发票。问题是用户表达方式非常多，规则很难覆盖完整。

大模型的思路不同。它不是靠人把所有规则一条一条写死，而是通过阅读海量文本数据，从书籍、网页、论坛、代码、百科等内容中学习语言规律和知识。训练完成后，它能根据上下文理解不同表达背后的真实意图。

我的理解：大模型的训练方式可以简单理解为“让机器读大量文本”。它不是靠人工规则工作，而是靠读了足够多的文本后，从数据中总结出语言如何运作。

2. 大模型训练阶段

大模型不是一开始就会像聊天助手一样回答问题。它通常会经历几个阶段，能力是一层一层训练出来的。

阶段 目标 产物/效果
预训练（Pre-training） 让模型阅读海量文本，学习语言规律、常识、代码和知识关联 得到基座模型（Base Model），主要能力是文本续写
指令微调（Instruction Tuning / SFT） 用“问题-答案”“任务-回复”样本教模型听懂人类指令 模型开始能按要求回答问题、总结、分类、写代码
对齐训练（Alignment） 让模型回答更有帮助、更安全、更符合人类偏好 得到更像助手的 Chat 模型或 Instruct 模型
领域微调/继续训练 用某个行业或业务数据继续训练模型 模型更熟悉特定领域表达和业务知识

可以这样理解：预训练阶段让模型“读了很多书”，指令微调让模型“学会按题目作答”，对齐训练让模型“更像一个会沟通的助手”。所以我们在 RAG 项目里通常调用的是已经完成指令微调和对齐训练的 Chat 模型，而不是只会续写文本的 Base 模型。

这里还要区分一个点：RAG 本身通常不是训练模型。RAG 是在提问时先从知识库检索相关资料，再把资料和问题一起交给 Chat 模型回答。也就是说，RAG 更像是给模型“临时开卷考试资料”，而不是把这些知识永久训练进模型参数里。

3. 大模型的“大”指参数量

大模型里的“大”，主要指参数量大。常见的 7B、14B、72B 中，B 是 Billion，表示十亿参数。参数可以粗略理解成模型内部存储知识和语言规律的连接权重。

参数越多，模型通常能力越强，但运行成本也更高，需要更多显存和算力。对 RAG 问答这类场景来说，并不是参数越大越好，很多时候 14B、32B 级别模型已经够用。选模型要看业务场景、响应速度、成本和部署方式。

4. Token 是大模型的计量单位

大模型处理文本不是按“字”或“词”直接计算，而是按 Token。Token 可以理解为模型把文本切分后的最小处理片段。

Token 重要的原因有两个：

  • API 调用通常按输入 Token 和输出 Token 计费。
  • 上下文窗口也是按 Token 计算，决定了一次能塞给模型多少内容。

所以后续做 RAG 时，不能无限制地把整本文档都塞给模型，而要先检索出最相关的片段，再交给模型回答。

5. 上下文窗口决定模型一次能看多少内容

上下文窗口指模型一次对话中能看到的文本总量上限，包括系统提示词、历史对话、当前问题、检索片段和模型输出。

这个概念对 RAG 很关键。如果把大量无关材料都塞进上下文，不仅可能超出窗口限制，也会让模型抓不住重点。RAG 的价值就在于先从知识库中检索最相关内容，只把关键片段放进上下文，让模型基于这些内容组织答案。

6. Temperature 控制回答的随机性

Temperature 是调用大模型时常见的参数，用来控制回答的稳定程度和创造性。

场景 推荐取值 理由
RAG 问答 0 到 0.3 要求基于资料准确回答，少发挥
代码生成 0 到 0.2 代码需要稳定和精确
日常聊天 0.5 到 0.7 需要自然、有变化
创意写作 0.7 到 1.0 需要更强的多样性

对 RAG 来说，答案应该来自检索到的资料，所以 Temperature 通常设置得比较低。

7. MoE 是混合专家架构

MoE 全称是 Mixture of Experts，混合专家模型。它的特点是模型总参数很多，但每次推理只激活其中一部分专家参数。

例如某些模型可能总参数是几百 B，但每次实际激活的参数只有几十 B。这样可以在保持较强能力的同时降低推理成本。作为应用开发者，不需要深入实现细节，只要知道看到“总参数/激活参数”时，实际推理成本主要看激活参数。

8. 主流模型要建立“模型地图”

文章梳理了国际和国内常见大模型。国际模型包括 GPT、Claude、Gemini、Llama 等；国内模型包括 DeepSeek、Qwen、GLM、MiniMax、Kimi 等。

对当前 Ragent/RAG 学习来说，重点关注 DeepSeek 和 Qwen：

  • DeepSeek：推理能力和性价比突出，适合理解 reasoning、Agent、复杂规划等方向。
  • Qwen：中文能力稳定，模型尺寸覆盖广，开源生态和工具链比较完整，适合国内业务和学习实践。

9. RAG 系统实际调用的是 Chat 模型

大模型可以分为基座模型和 Chat 模型。

基座模型来自预训练阶段，主要能力是续写文本。它读过大量文本，但不一定会按照人的指令回答问题。

Chat 模型是在基座模型基础上经过指令微调、对齐训练等步骤得到的模型。它更擅长理解“你要它做什么”，并按照用户问题、系统提示词和上下文材料生成答案。

RAG 的最后一步是“根据检索片段回答用户问题”，这需要模型理解指令和上下文，所以 RAG 系统通常调用 Chat 模型，而不是 Base 模型。

常见模型后缀含义：

后缀 含义 示例
-Instruct 经过指令微调的 Chat 模型 Qwen2.5-7B-Instruct
-Chat 和 Instruct 类似，强调对话能力 GLM-4-Chat
-GPTQ / -AWQ 量化版本，模型体积更小，适合低资源部署 Qwen2.5-7B-Instruct-GPTQ-Int4

看到 -Instruct 或 -Chat，一般可以认为这个模型更适合问答、对话、摘要、RAG 这类任务；看到 -GPTQ、-AWQ、Int4 等字样，说明它通常是为了降低显存占用做过量化处理的版本。

10. 量化是为了降低本地部署成本

量化是本地部署模型时非常重要的概念。它的核心目的就是：在尽量少损失模型效果的前提下，让模型文件更小、显存占用更低、推理成本更可控。

10.1 模型参数本质上是权重数字

大模型的参数本质上是一堆数字，也叫权重。模型输入一句话后，会把文本切成 Token，然后经过很多层计算，最后判断下一个 Token 最可能是什么。

可以把模型想象成一个巨大的打分系统。每一层计算里用到的乘法系数、加法偏移量，都是模型参数。例如模型判断“今天天气”后面更可能接“晴”还是“雨”，内部会对不同候选词计算得分，得分高的词更可能被输出。

所谓 7B 模型，就是模型里大约有 70 亿个这样的参数。训练时这些参数会不断被调整；训练完成后，它们会固定下来，保存成模型权重文件。

10.2 FP32、FP16、BF16 是什么

模型权重是数字，数字需要用某种精度格式保存。常见格式有 FP32、FP16、BF16、TF32。

浮点数大致由三部分组成：

  • 符号位：表示正数还是负数。
  • 指数位：决定能表示多大或多小的数字，也就是数值范围。
  • 尾数位：决定小数能表示多精细，也就是精度。

可以用“快递地址”来理解：指数位像省市信息，决定能送多远；尾数位像门牌号，决定能写多细。总空间有限，指数位多，范围更大；尾数位多，精度更细。

格式 位数 每个参数占用 特点
FP32 32 位 4 字节 传统全精度，范围大、精度高，但太占显存
TF32 19 位 通常由 GPU 内部处理 NVIDIA Ampere 架构引入，保留 FP32 的范围，降低部分精度，开发者一般不用管
FP16 16 位 2 字节 半精度，体积是 FP32 的一半，精度较好，但数值范围较小，训练时容易溢出
BF16 16 位 2 字节 脑浮点，范围接近 FP32，精度比 FP16 粗一些，但训练更稳定

FP16 和 BF16 都是 16 位，但分配方式不同：

  • FP16：更偏向保留精度，数值范围较小。
  • BF16：更偏向保留范围，精度稍低，但稳定性更好。

对大模型来说，训练和推理过程中数值范围可能变化很大，所以 BF16 很常见。很多模型在 Hugging Face 上的原始权重默认就是 BF16。平时说“原始精度”或“未量化”，很多时候指的是 BF16，而不是 FP32。

10.3 什么是量化

量化就是把模型权重从 BF16/FP16 这种 16 位精度，进一步压缩成更低位数，例如 INT8 或 INT4。

精度降低后，每个参数占用空间变小，模型文件和显存占用也会变小。

精度 每个参数占用 7B 模型权重大小估算 说明
FP32 4 字节 约 28 GB 全精度，当前很少直接用来存模型权重
BF16 / FP16 2 字节 约 14 GB 常见原始权重精度
INT8 1 字节 约 7 GB 轻度量化，效果损失较小
INT4 0.5 字节 约 3.5 GB 常见量化选择，体积更小，但效果会有一定损失

注意：表格里只是模型权重本身的估算。实际运行时还需要额外显存保存 KV Cache、计算中间结果等，所以真实显存占用会更高。一般要预留权重大小 1.2 到 1.5 倍左右的显存。

10.4 常见量化格式

在下载模型或查看模型后缀时，经常会看到 GPTQ、AWQ、GGUF 这些格式。

格式 全称 特点 适合场景
GPTQ GPT Post-Training Quantization 经典的训练后量化方案，社区支持广，通常用于 GPU 推理 有 NVIDIA 显卡，想要较好的量化效果
AWQ Activation-aware Weight Quantization 激活感知权重量化，通常速度和效果都比较好 有 NVIDIA 显卡，优先考虑的新一些的 GPU 量化方案
GGUF GPT-Generated Unified Format llama.cpp 生态常见格式，支持 CPU，也支持部分 GPU 加速 没有独显、显存不足，或想在本地轻量运行

简单记法：

  • 有 NVIDIA 显卡：优先看 AWQ 或 GPTQ。
  • 显卡不够或想用 CPU 跑：优先看 GGUF。

10.5 量化等级怎么选

以 GGUF 为例，经常会看到 Q4_K_M、Q5_K_M、Q8_0 这样的名字。这里的数字大致表示量化位数，数字越大，精度越高，文件越大，显存占用也越高。

量化等级 含义 特点
Q4_K_M 4 位量化 体积小，显存压力低，性价比高，常用选择
Q5_K_M 5 位量化 在效果和体积之间更平衡
Q8_0 8 位量化 接近原始精度，但文件和显存占用更大

普通本地部署时，INT4 或 Q4_K_M 往往已经够用。如果对输出质量要求更高，或者硬件资源更充足，可以考虑 Q5_K_M 或 Q8_0。

10.6 当前阶段需要关心到什么程度

如果是通过 SiliconFlow、OpenAI、DeepSeek API、通义千问 API 这类平台调用模型，暂时不用关心量化。平台已经处理了部署、显存、推理优化等问题，我们只需要关注模型名、输入输出、上下文窗口、价格和效果。

只有在本地部署模型时，才需要根据自己的硬件选择量化版本。这个时候要重点看三件事：模型参数量、量化格式、量化等级。

我的阶段性理解：

  • BF16/FP16：更接近原始模型效果，但显存占用大。
  • INT8：效果损失较小，体积减半。
  • INT4：显存压力小，适合个人电脑尝试本地部署，但回答质量可能略有下降。
  • GPTQ/AWQ/GGUF：不是模型能力本身，而是模型权重的量化/存储/推理格式。
  • 对 RAG 项目学习来说，先用 API 跑通流程更重要，本地量化部署可以后面再深入。

11. 深度思考适合推理，但 RAG 通常不需要

深度思考通常和 CoT（Chain of Thought，思维链）有关，适合数学推理、复杂规划、代码分析等需要多步推理的任务。

但在普通 RAG 问答里，答案应该已经存在于检索到的文本片段中。模型主要负责理解、提取和表达，不需要展开很长的推理过程。因此 RAG 场景通常不需要开启深度思考，否则会增加 Token 消耗和响应时间。

12. 开发者需要 API，而不是网页端

网页端适合个人聊天和体验模型能力，但不适合接入业务系统。它的问题包括不能集成到 Java 系统、不能批量处理、不能稳定控制参数、不能自定义完整提示词流程，也很难实现 RAG 的“检索后生成”链路。

API 调用则可以把大模型当成一个远程函数：系统把模型名、系统提示词、用户问题、Temperature 等参数发过去，模型返回回答。这样大模型就能成为业务系统的一部分。

本系列后续会使用 SiliconFlow 平台和 Qwen 系列模型，原因是国内访问方便、注册简单、支持多种开源模型，并且 API 兼容 OpenAI 协议，后续切换平台时代码改动较小。

重点回顾

  • 大模型不是靠人工写死规则工作，而是从海量文本中学习语言规律。
  • 大模型训练通常包括预训练、指令微调、对齐训练等阶段，最终才形成适合对话和任务执行的 Chat 模型。
  • 参数量影响模型能力和成本，但不是越大越好，要根据场景选择。
  • Token、上下文窗口、Temperature 是后续调用 API 时必须理解的基础概念。
  • RAG 场景下应优先使用 Chat 模型，并把 Temperature 设置得较低。
  • API 调用是把大模型接入 Java/RAG 系统的关键方式。

易混点整理

  • Base Model 不是 Chat Model。Base Model 主要会续写文本，Chat Model 才更适合对话、问答、摘要和 RAG。
  • RAG 通常不是训练模型，而是在提问时检索资料，再把资料交给模型参考。
  • Token 不是字数，也不是严格意义上的词数，而是模型内部的文本切分单位。
  • 上下文窗口不是只算用户问题，而是系统提示词、历史对话、检索片段和模型输出都要占窗口。
  • Temperature 越高不代表模型越聪明，只代表回答更随机、更有创造性。
  • 参数量越大不一定越适合业务，速度、成本、上下文窗口、中文能力和工具调用能力都要一起考虑。
  • GPTQ、AWQ、GGUF 不是模型类型，而是量化/存储/推理相关格式。
