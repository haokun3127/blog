论文标题 Recommendation as Instruction Following: A Large Language Model Empowered Recommendation Approach
期刊 未在原笔记中注明
发表日期 2023-05-11
作者 Junjie Zhang；Ruobing Xie；Yupeng Hou；Wayne Xin Zhao；Leyu Lin；Ji-Rong Wen
核心主题 将推荐任务重构为“指令遵循”任务
关键词 InstructRec、Preference、Intention、Task Form、Instruction Tuning、Reranking
原文 PDF Zhang 等 - 2023 - Recommendation as Instruction Following A Large Language Model Empowered Recommendation Approach.pdf
笔记创建日期 2025/12/9 18:45:05

摘要速览

传统的推荐模型主要依赖 ID 特征拟合用户偏好，虽然有效，但在冷启动和跨域场景中的泛化能力有限。本文提出 InstructRec，将推荐任务重构为“指令遵循”任务，让用户偏好、即时意图和任务形式都能用自然语言表达，并基于 25.2 万条个性化指令数据对 Flan-T5-XL 进行微调。实验表明，该方法在顺序推荐、产品搜索和个性化搜索等任务上优于传统基线以及未微调的 GPT-3.5。

项目 内容
英文摘要 In the past decades, recommender systems have attracted much attention in both research and industry communities, and a large number of studies have been devoted to developing effective recommendation models. Basically speaking, these models mainly learn the underlying user preference from historical behavior data, typically in the forms of item IDs, and then estimate the user-item matching relationships for recommendations. Inspired by the recent progress on large language models, this paper considers recommendation as instruction following by LLMs. The authors design a general instruction format, build 39 instruction templates, automatically generate 252K user-personalized instructions, and fine-tune Flan-T5-XL to better adapt LLMs to recommender systems. Experiments on several recommendation and search tasks show that the proposed approach outperforms multiple competitive baselines, including GPT-3.5.
中文摘要翻译 传统推荐模型主要依赖 ID 特征拟合用户偏好，虽然有效但面临冷启动和泛化能力差的问题。受 LLM 进展启发，本文提出 InstructRec，将推荐任务重构为“指令遵循”任务。核心思想是用户的偏好和需求可以用自然语言指令描述。作者设计了通用的指令格式，并基于 GPT-3.5 生成 252K 条个性化指令数据，对 Flan-T5-XL（3B）进行微调。实验表明，该方法在顺序推荐、搜索等任务上优于传统基线及未微调的 GPT-3.5，展示了更好的泛化性和用户交互能力。

1. 文章基本框架

背景

传统的推荐系统主要依赖于协同过滤或深度神经网络，通过拟合用户和物品 ID 的历史交互数据来捕捉偏好。

  • ID 范式的局限：传统模型（如 SASRec、LightGCN）高度依赖 ID Embeddings。虽然在闭环数据内有效，但缺乏语义信息，导致在冷启动和跨域推荐场景下表现较差。
  • 交互范式被动：传统推荐是“系统猜用户”，用户是被动的接收者。现有的 LLM + Rec 尝试往往缺乏灵活性，用户无法主动通过自然语言表达多样化需求。
  • 语义鸿沟：LLM 懂通用语言，但并不天然理解推荐系统中的协同信号。

目的

本研究的核心目的是解决传统推荐系统长期面临的泛化能力差与交互被动两大痛点，并探索大语言模型在推荐领域的深度应用潜力。

  • 打破 ID 范式的局限：传统推荐模型高度依赖 ID Embeddings，这种方式虽然在闭环数据内有效，但缺乏语义信息，导致在冷启动和跨域推荐场景下表现较差。
  • 改变交互范式：让用户可以通过自然语言指令主动表达偏好和需求，而不是只能被动接受推荐结果。
  • 弥合 LLM 与 RecSys 的鸿沟：通过指令微调，将 LLM 改造为既懂通用语言又懂个性化推荐逻辑的“推荐专家”。

结论

通过在 3B 参数量的 Flan-T5-XL 模型上进行指令微调，InstructRec 在顺序推荐、产品搜索和个性化搜索等多个任务上均优于传统强基线以及未微调的 GPT-3.5。实验证明，该方法能有效增强 LLM 适应多样化用户需求的能力，并在跨数据集测试中表现出良好的泛化性。

2. 结果与讨论

数据以及数据来源

  1. 原始数据集：使用 Amazon Review 数据集中的 Video Games 作为主要实验数据，CDs & Vinyl 作为跨域泛化测试数据。
  2. 指令数据：作者利用 GPT-3.5 作为 Teacher LLM，基于原始交互数据生成了 25.2 万条细粒度的个性化指令数据。
  3. 模板来源：这些数据覆盖 39 种粗粒度模板，其中约 15.1 万条涉及用户偏好，约 10.1 万条涉及决策意图。

方法

论文提出了 InstructRec 框架，其核心在于将推荐任务重构为“指令遵循”任务。具体可以分为三个关键部分。

推荐指令格式的形式化定义（Instruction Format）

为了让 LLM 理解推荐任务，作者设计了一套通用的指令模板结构 Instruction = <P, I, T>，包含三个关键维度：

  • 偏好（Preference, P）：描述用户的长期兴趣。

    1. P_0：无偏好（None / Cold-start）

    • 定义：系统中完全没有任何关于用户的历史行为或画像信息。
    • 对应场景：新用户冷启动或用户关闭历史追踪的隐私模式。
    • 模型处理方式：
      • 指令中的 P 部分为空。
      • 模型只能依赖意图（如果用户提供了）或上下文进行推荐。
      • 如果连意图都没有，模型通常会退化为推荐全局热门物品。
    • 指令示例：

      “Help me find a gift.”

    2. P_1：隐式偏好（Implicit Preference）

    • 定义：通过用户历史交互记录间接反映出的偏好。
    • 数据来源：用户的点击、购买等隐式反馈序列。
    • InstructRec 的处理方式：
      • 传统做法会直接使用一串物品 ID。
      • 论文将其转化为物品标题的文本序列，让 LLM 能利用语义理解能力捕捉物品之间的关系。
    • 指令示例：

      “The user has previously purchased: Resident Evil 2, Silent Hill, Outlast…”

    3. P_2：显式偏好（Explicit Preference）

    • 定义：用户直接用自然语言描述的兴趣画像，是对历史行为的抽象总结。
    • 数据来源与创新点：
      • 现实中用户很少主动写出完整兴趣画像。
      • 作者利用 GPT-3.5 对 P_1 进行总结，自动生成 P_2。
    • 构建过程：
      • Input to GPT-3.5: “User bought: Resident Evil 2, Silent Hill, Outlast. Infer the user’s preference.”
      • Output (P_2): “The user prefers horror-survival games with strong narrative elements.”
    • 价值：P_2 是对 P_1 的语义升维，有助于模型理解更抽象、更高级的兴趣特征。

    总结对比

    符号 类型 数据形式 语义层级 生成方式 适用场景
    P_0 无偏好 空 无 - 冷启动用户
    P_1 隐式偏好 物品标题序列 低 直接查库获取 常规推荐
    P_2 显式偏好 自然语言画像 高 GPT-3.5 推理生成 高级个性化推荐

    理解这三者的区别，关键在于论文如何从 P_1 出发，利用大模型生成 P_2，从而让 InstructRec 同时理解具体行为模式和抽象兴趣偏好。

  • 意图（Intention, I）：描述用户当前的即时需求。

    • I_0：无明确意图，对应传统的“猜你喜欢”或序列推荐。
    • I_1：模糊意图，如“找点适合送礼的东西”。
    • I_2：明确意图，如“256G 蓝色手机”。

  • 任务形式（Task Form, T）：定义模型需要执行的操作。

    • T_0：Pointwise，判别式推荐，输出 Yes / No。
    • T_1：Pairwise，对比推荐。
    • T_2：Matching，生成式召回。
    • T_3：Reranking，对候选列表进行重排序。

大规模指令数据的自动化构建（Instruction Generation）

由于现有数据集缺乏自然语言指令，作者采用了 Self-Instruct 策略，利用 GPT-3.5 作为教师模型来合成数据：

  1. 模板设计：手工设计了 39 个通用指令模板，覆盖从序列推荐到个性化搜索的多种场景。
  2. 语义标注与合成：
    • 生成 P_2：让 GPT-3.5 根据用户历史购买记录总结兴趣画像。
    • 提取 I_1：利用用户评论信息，让 GPT-3.5 推断用户当时的购买动机。
  3. 增强多样性策略：
    • 任务反转：例如已知用户购买某物，反推他当时可能搜索了什么。
    • 思维链：要求模型先生成推理过程，再给出推荐结果，以增强可解释性。

通过上述流程，作者基于 Amazon Reviews 数据集构建出了 25.2 万条高质量细粒度指令数据。

基于指令的大模型微调（Instruction Tuning）

  • 基座模型：选择 Flan-T5-XL（3B 参数），Encoder-Decoder 架构更适合处理长输入。
  • 训练方式：全参数监督微调。
  • 推理模式：主要采用重排序模式，即先由轻量级模型召回候选集，再由 InstructRec 做精排，以平衡效果与效率。

结论分析

  1. 顺序推荐（P_1 + I_0 + T_3）：InstructRec 在 HR@1 和 NDCG@5 等指标上均超越了 SASRec 和 BERT4Rec。值得注意的是，未微调的 GPT-3.5 表现最差，证明了私域数据微调的必要性。
  2. 个性化搜索（P_1 / P_2 / I_1 / I_2 + T_3）：这是文章的亮点。InstructRec 在处理模糊意图和显式偏好时表现出色，大幅领先基线模型 TEM，例如在 HR@1 上提升 39.04%。
  3. 跨域泛化：在 Games 数据集训练后直接在 CDs 数据集测试。虽然性能不如域内训练的传统模型，但明显优于 Zero-shot 的 GPT-3.5 和原始 Flan-T5，证明其学到了更通用的推荐逻辑。

3. 文章亮点思考

1. 本文有什么优缺点？

  • 优点
    • 范式创新：将推荐问题形式化为指令遵循问题，统一了搜索、推荐、重排序等多种任务形式，<P, I, T> 的抽象很有普适性。
    • 通用指令协议：提出了一套能够统一序列推荐、搜索、解释等多种任务的标准化数据格式，为后续 RecSys + LLM 研究提供了通用接口。
    • 数据构造巧妙：利用 Teacher LLM 从评论和行为中蒸馏出显式偏好与意图，缓解了推荐领域缺乏自然语言 Ground Truth 的问题。
  • 缺点 / 疑问
    • 上下文长度限制：Flan-T5 只有 512 token 的限制，导致用户行为序列必须截断，长序列信息会丢失。
    • 主要用于重排：由于推理成本高，InstructRec 更适合作为重排器，而不是全库召回模型，工业落地时可能面临延迟压力。
    • 幻觉问题：虽然通过候选集约束缓解了一部分问题，但在生成式任务中，大模型仍可能输出不在候选列表中的物品。

2. 可以如何改进？

  1. 扩展上下文窗口：使用支持长上下文的模型，或外挂记忆机制处理更长的用户历史序列。
  2. 多轮对话交互：从单轮指令扩展到多轮对话，让系统在意图模糊时主动追问用户。
  3. 更强的基座模型：用 Llama 3、Mistral 等更强的开源模型替代 Flan-T5，可能会进一步提升效果。

3. 这篇文章为什么值得关注？

这篇文章的价值在于它系统性地桥接了通用大模型能力与垂直推荐场景：

  1. 它清晰定义了“指令”在推荐中的含义，不只是把推荐数据硬塞给 LLM，而是拆解成偏好、意图和任务三个维度。
  2. 它用实验验证了 Recommendation + Instruction Tuning 这条路线的有效性，说明“直接拿 GPT-3.5 做推荐”并不可行，领域微调才是关键。
  3. 它让模型通过自然语言理解用户与物品，从而摆脱了纯 ID 推荐范式在冷启动和跨域场景中的限制。
