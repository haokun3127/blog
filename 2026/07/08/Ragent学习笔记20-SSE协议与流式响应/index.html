Ragent学习笔记20：SSE 协议与流式响应

学习导览

大模型流式输出看起来像“一个字一个字蹦出来”，背后常用的协议就是 SSE。

SSE 的全称是 Server-Sent Events，本质是服务端通过一个 HTTP 长连接持续向客户端推送事件。它比 WebSocket 简单，特别适合“客户端发起请求，服务端持续返回结果”的场景。

知识速览

字段 作用 说明
data: 事件数据 最常用，承载返回内容
event: 事件类型 可以区分 message、error、done 等
id: 事件 ID 支持断线重连后续传
retry: 重连间隔 告诉浏览器多久后重连
: 注释行 心跳 保持连接不被代理断开

详细笔记

SSE 是什么

SSE 是一种基于 HTTP 的服务端推送机制。客户端建立连接后，服务端可以不断往连接里写数据。

常见格式：

每个事件之间用空行分隔。

data: 字段

data: 是最核心的字段。大模型流式响应中，每个 chunk 通常都会放在 data: 后面。

如果一个事件有多行 data:，客户端会把它们合并成一个事件数据。

event: 字段

event: 用来定义事件类型。默认类型是 message。

id: 和 retry:

id: 可以让客户端记录最后收到的事件 ID。断线重连时，浏览器会带上 Last-Event-ID。

retry: 用来告诉浏览器重连间隔。

不过大模型生成通常是一次性流，不一定需要严格续传；但在业务事件流里，id 很有价值。

SSE、WebSocket、长轮询对比

方案 通信方向 优点 适合场景
SSE 服务端 -> 客户端 简单，基于 HTTP，自动重连 大模型流式输出、通知流
WebSocket 双向 实时双向通信能力强 IM、协作编辑、游戏
长轮询 伪实时 兼容性好 简单通知

大模型 API 选择 SSE 的原因很直接：用户发起一次请求后，主要是服务端持续返回内容，不需要频繁双向通信。

大模型流式 chunk

非流式响应是一次性返回完整答案；流式响应则是多次返回 chunk。

每个 chunk 里可能包含：

解析时不能假设每个 chunk 都有内容，有些 chunk 可能只是角色信息、结束标记或统计信息。

Java 客户端要注意什么

生产级 SSE 客户端不能只做字符串 split。要考虑：

  1. 网络超时。
  2. 连接中断。
  3. 空 delta。
  4. 多行 data。
  5. [DONE] 结束标记。
  6. 异常事件。
  7. Token 统计可能只在最后返回。

粘包问题

SSE 是基于字节流的，底层读取时可能一次读到半个事件，也可能一次读到多个事件。

所以客户端要按 SSE 的事件边界解析，也就是用空行作为事件结束，而不是简单认为“一次 read 就是一条消息”。

大模型流式响应常用 SSE，因为它适合一次请求后服务端持续推送内容的场景，比 WebSocket 更轻量。SSE 每个事件由 data、event、id、retry 等字段组成，事件之间用空行分隔。后端解析模型流时要处理空 delta、结束标记、网络中断、超时、多行 data 和粘包问题，不能简单把一次网络读取当成一个完整事件。
