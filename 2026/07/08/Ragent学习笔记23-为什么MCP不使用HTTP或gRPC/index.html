Ragent学习笔记23：为什么 MCP 不使用 HTTP 或 gRPC

发表于2026-07-08|更新于2026-07-08|Ragent

学习导览

这个问题容易混淆：MCP 不是完全“不用 HTTP”，而是不把 HTTP 当成自己的调用协议。

HTTP 可以作为传输层，负责把数据送过去；JSON-RPC 负责定义这条消息到底是请求、响应、通知还是错误。

知识速览

技术 定位 特点
HTTP 传输协议 负责请求和响应的传输，不规定 RPC 消息语义
gRPC 强类型 RPC 框架 性能强、IDL 清晰，但较重
JSON-RPC 2.0 轻量 RPC 消息协议 人类可读、实现简单、传输无关
MCP 上下文协议 基于 JSON-RPC 定义工具、资源、Prompt 等交互

详细笔记

HTTP 是传输协议，不是调用协议

HTTP 规定了请求方法、Header、Body、状态码等，但它不直接规定：

这些是 RPC 消息协议要解决的问题。

所以 MCP 可以运行在 HTTP 之上，但仍然需要 JSON-RPC 来统一消息结构。

gRPC 为什么不适合 MCP

gRPC 是很成熟的 RPC 框架，性能强、类型清晰，但它对 MCP 来说偏重。

主要问题：

  1. 需要 .proto 文件和代码生成。
  2. 对浏览器和轻量客户端不够友好。
  3. 二进制协议不如 JSON 易读易调试。
  4. 多语言生态虽然强，但接入门槛比 JSON-RPC 高。
  5. MCP 需要灵活暴露工具、资源和 Prompt，JSON 结构更轻便。

MCP 的目标是让各种 Host 和工具服务快速互联，而不是先引入一套较重的 RPC 工程体系。

JSON-RPC 的优势

JSON-RPC 适合 MCP 的原因：

优势 说明
统一消息格式 Request、Response、Notification、Error 都有标准结构
轻量易实现 不依赖代码生成，普通 JSON 库即可处理
人类可读 排查问题时可以直接看消息内容
传输无关 可以跑在 Stdio、SSE、HTTP 等传输上
支持通知 能表达不需要响应的单向消息
支持批量调用 可以一次发送多个请求

这些特性和 MCP 的需求很契合。

JSON-RPC 也不是万能的

JSON-RPC 只解决消息格式，不解决所有工程问题。

它不包含：

  1. 鉴权机制。
  2. 传输层细节。
  3. 类型系统。
  4. 服务发现。
  5. 限流熔断。
  6. 链路追踪。

所以企业级 MCP 服务仍然要自己补齐安全、稳定性和可观测性。

MCP、HTTP、JSON-RPC 的关系

可以用一层一层的方式理解：

这也是为什么说“MCP 不直接使用 HTTP 或 gRPC”，更准确地说是：MCP 选择 JSON-RPC 作为调用消息协议，HTTP 只是可选传输方式之一。

复习抓手

这篇可以记住：

面试回答可以这样说：

MCP 没有直接把 HTTP 当成调用协议，因为 HTTP 只规定传输，不规定 RPC 方法、请求 ID、通知、错误对象等调用语义。gRPC 虽然是完整 RPC 框架，但对 MCP 这种需要轻量、多语言、易调试、传输无关的协议来说偏重。JSON-RPC 2.0 用 JSON 定义 Request、Response、Notification、Error，简单可读、容易实现，还能运行在 Stdio、SSE、Streamable HTTP 等不同传输上，所以更适合作为 MCP 的消息协议。
