Ragent学习笔记17：查询重写与语义增强机制

学习导览

用户说的话，往往不是最适合检索的词。

比如用户问：

人可以通过上下文知道“它”指的是某个商品或服务，但检索系统看到的可能只有“它”“退款”，缺少明确主体。直接拿这个问题去向量检索，很可能召回不到正确内容。

Query Rewrite 的目标就是把用户问题改写成更适合检索的查询。

知识速览

策略 解决的问题 示例
指代消解 “它”“这个”“刚才那个”指代不明 “它怎么退货” -> “iPhone 16 Pro 如何退货”
上下文补全 当前问题缺少上下文 “保修多久” -> “购买的空调保修多久”
口语转正式 用户表达太口语 “坏了咋办” -> “商品故障后的售后处理流程”
多意图拆分 一个问题包含多个子问题 “能退吗，多久到款” -> 退货政策 + 退款到账时间
关键词扩展 检索词太窄 “退款” -> 退款、退货、售后、原路返回

查询重写在 RAG 中的位置

查询重写位于“用户原始问题”和“检索召回”之间。它不是为了让最终回答更好听，而是为了让检索阶段更容易找到正确材料。

详细笔记

检索系统没有对话记忆

多轮对话里，模型可以通过 messages 看到历史上下文，但检索系统通常只拿当前 query 去查向量库。

这会导致一个问题：

所以用户第二轮问“那这个支持七天无理由吗”，模型也许知道“这个”指什么，但向量检索只看到一句不完整的话。

Query Rewrite 的价值，就是把上下文补到检索 query 里。

指代消解

指代消解是把代词还原成明确对象。

这一步非常关键，因为向量库里不会存“它”的含义，只会存具体文档内容。

上下文补全

有些问题不是指代不明，而是省略了上下文。

补全后的 query 更像一条完整的搜索句，召回效果通常更稳定。

口语化转正式

用户表达可能很口语：

知识库文档里可能写的是：

如果不做改写，口语表达和文档表达之间可能存在语义距离。口语转正式能让 query 更贴近知识库的表达方式。

多意图拆分

一个用户问题可能包含多个意图：

它实际上包含两个检索目标：

  1. 退货条件。
  2. 退款到账时间。

如果只做一次检索，可能召回结果偏向其中一个子问题。拆分后可以分别检索，再融合结果。

关键词扩展

关键词扩展适合解决“用户词太窄”的问题。

比如用户说“退款”，知识库可能写“退货”“售后”“原路返回”“退款到账”。适当扩展关键词可以提升召回率。

但扩展也要克制。扩展太多会引入噪声，让召回结果变散。

用大模型做 Query Rewrite

用大模型改写 query 时，Prompt 要强调几点：

要求 说明
不回答问题 只改写，不生成最终答案
保留用户意图 不添加用户没问的新目标
补全必要上下文 从历史对话里提取主体和约束
可拆分多意图 多个问题拆成多个 query
输出结构化 方便后续程序解析

成本与时机

Query Rewrite 需要额外调用一次模型，所以不是所有问题都必须改写。

RAG 系统里，检索阶段通常只拿当前 query 去查知识库，因此多轮场景会出现指代不明、上下文缺失、口语表达和多意图混杂等问题。Query Rewrite 会在检索前对用户问题做指代消解、上下文补全、口语转正式、多意图拆分和关键词扩展，让检索 query 更完整、更贴近知识库表达，从而提升召回质量。
