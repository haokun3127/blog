Ragent学习笔记19：RAG 模型检索生成评估与优化

发表于2026-07-08|更新于2026-07-08|Ragent

  • 《AI大模型Ragent项目》——RAG模型检索生成评估与优化
  • 本地文档路径：C:\Users\ASUS\Desktop\项目代码\ragent\文档\《AI大模型Ragent项目》——RAG模型检索生成评估与优化.html

学习导览

RAG 系统不能只靠“感觉还不错”来判断效果。因为答案不好时，问题可能出在很多地方：

所以 RAG 评估要分层看：检索阶段看召回质量，生成阶段看答案质量，端到端看用户最终体验。

知识速览

层次 指标 关注点
检索阶段 Hit@K 前 K 条里有没有命中答案材料
检索阶段 MRR 正确材料排得靠不靠前
检索阶段 Recall / Precision 召回完整性和准确性
生成阶段 Faithfulness 答案是否忠实于上下文
生成阶段 Answer Relevancy 答案是否回答了问题
端到端 答案正确率 用户最终拿到的答案对不对
端到端 兜底率 系统不能回答时是否合理拒答
端到端 用户满意度 真实用户反馈

分层评估思路

评估时不能只看最终答案，否则很难定位问题。要把链路拆开看，才能知道应该优化切分、检索、重排序、Prompt 还是生成策略。

详细笔记

为什么不能靠感觉优化

如果只靠人工感受，容易出现两个问题：

  1. 样本太少，容易被少数好答案误导。
  2. 无法定位问题，只知道“不好”，不知道坏在哪。

比如一个答案错了，可能是检索阶段没找到材料，也可能是材料找到了但模型乱编。两种问题的优化方向完全不同。

检索阶段指标

检索阶段的目标是把正确材料找出来，并尽量排在前面。

Hit@K

Hit@K 表示前 K 条检索结果里是否包含正确答案材料。

它适合判断召回链路有没有把答案材料找出来。

MRR

MRR 关注正确答案排在第几位。正确文档排第 1 位，比排第 5 位更好。

MRR 能反映排序质量。

Recall 与 Precision

Recall 看该找的有没有找全，Precision 看找出来的结果里有多少是有用的。

在 RAG 里，召回率不足会导致模型没有依据；精确率不足会让 Prompt 混入噪声。

生成阶段指标

生成阶段关注模型有没有基于上下文回答。

指标 含义
Faithfulness 答案是否忠实于检索上下文
Answer Relevancy 答案是否和用户问题相关

一个答案可能看起来很流畅，但如果上下文里没有依据，就是幻觉风险。

端到端指标

端到端指标看最终用户体验：

最终要看答案是否正确、是否完整、是否该拒答时拒答。

评测集构建

评测集通常包含这些字段：

字段 说明
question 用户问题
expected_answer 标准答案或参考答案
expected_context 应该命中的文档或 chunk
category 问题类型
difficulty 难度

评测集来源可以是人工标注、用户反馈，也可以由大模型辅助生成后再人工校验。

LLM-as-Judge

LLM-as-Judge 是用大模型给答案评分。它适合快速批量评估，但不能完全替代人工。

需要注意：

  1. 评分 Prompt 要稳定。
  2. 要抽样人工校准。
  3. 不能让评测模型和被测模型互相“自嗨”。
  4. 对关键业务问题仍然要人工复核。

评估驱动优化

评估的价值在于指导优化：

评估现象 可能原因 优化方向
Hit@K 低 分块差、Embedding 差、查询没改写 优化 Chunk、Embedding、Query Rewrite
MRR 低 正确结果排太后 加 Rerank、调排序策略
Faithfulness 低 模型乱编 加引用约束、拒答策略
Answer Relevancy 低 Prompt 组织差 优化 Prompt 模板
兜底率异常 阈值不合理 调整相似度阈值和拒答条件

复习抓手

这篇重点记住：

面试回答可以这样说：

RAG 系统不能只看最终答案，需要分层评估。检索阶段用 Hit@K、MRR、Recall、Precision 判断是否召回正确材料以及排序质量；生成阶段用 Faithfulness 和 Answer Relevancy 判断答案是否忠实上下文、是否回答问题；端到端再看答案正确率、兜底率和用户满意度。通过评估结果定位问题，再针对 Chunk、Embedding、Query Rewrite、Rerank、Prompt 或拒答策略做优化。
