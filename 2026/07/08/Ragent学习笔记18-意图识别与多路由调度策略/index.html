Ragent学习笔记18：意图识别与多路由调度策略

学习导览

不是所有用户问题都应该进入 RAG 检索。

有些问题应该查知识库，有些应该调用工具，有些只是闲聊，还有些信息不完整，需要先追问。意图识别就是在问答链路前面加一个“分诊台”。

如果没有意图识别，系统就会变成“一根筋”：所有问题都检索知识库，检索不到就乱答。

知识速览

意图类型 含义 处理方式
知识检索 用户问知识库中已有信息 走 RAG 检索生成
工具调用 用户需要实时数据或业务操作 走 MCP/Function Call
闲聊对话 普通寒暄、非业务问题 直接模型回答或固定回复
引导澄清 信息不足，无法判断 先追问用户补充信息

四类意图

意图识别不是为了炫技，而是为了让不同类型的问题走不同的系统链路。

详细笔记

为什么需要意图识别

假设用户问：

这不是一个纯知识库问题。知识库里可能有“发货规则”，但用户真正需要的是某个订单的实时状态。这个时候只做 RAG 检索就不够，需要调用订单工具。

再比如用户问：

这类问题没有必要查知识库，可以直接回答。

所以意图识别解决的是“下一步该走哪条链路”的问题。

知识检索意图

知识检索适合回答规则、制度、说明文档类问题。

这类问题的答案通常在知识库中，可以走向量检索、Rerank、Prompt 组装和大模型生成。

工具调用意图

工具调用适合实时数据查询或业务动作。

这类问题需要外部系统参与，适合走 MCP 工具或 Function Call。

闲聊意图

闲聊不需要知识库，也不需要工具。可以直接由模型回答，或者走固定模板。

这样做可以减少不必要的检索和模型调用成本。

引导澄清意图

当用户问题缺少必要信息时，不应该硬查硬答。

这类问题应该先追问，而不是盲目调用工具。

三种识别方案

方案 优点 缺点 适用场景
规则匹配 快、成本低、可控 覆盖不全，维护成本高 明确关键词场景
大模型分类 语义理解强 成本高，结果可能不稳定 复杂表达场景
混合方案 兼顾效率和准确率 架构更复杂 生产推荐

生产系统通常会先用规则处理强确定性问题，再用大模型处理复杂语义，最后加兜底策略。

生产环境注意事项

意图识别上线后，需要持续观察：

指标 说明
分类准确率 是否把问题分到正确链路
兜底率 有多少问题无法判断
工具误调用率 是否把知识问题误判成工具调用
检索误触发率 是否不该查知识库却查了
用户追问率 澄清是否过多影响体验

意图边界不是一次设计就固定的，需要根据真实问题不断调整。

在 Agentic RAG 系统里，用户问题不一定都适合知识库检索。意图识别会先判断问题属于知识检索、工具调用、闲聊还是引导澄清，再由路由器分发到 RAG、MCP 工具、直接回复或追问链路。生产环境一般采用规则 + 大模型分类的混合方案，并通过准确率、兜底率、误调用率等指标持续优化。
