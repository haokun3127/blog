学习导览

大模型本身并不会自动记住前面聊过什么。每次请求对模型来说都是一次新的调用，所谓“记忆”，本质上是应用层把历史消息重新塞进 messages 里。

所以多轮对话设计的核心不是“让模型拥有记忆”，而是：

这就会引出几个问题：历史消息全带会爆 Token，不带又会失忆；只保留最近几轮可能丢掉长期背景；摘要压缩能省 Token，但可能丢细节。

知识速览

记忆策略 做法 优点 问题
完整历史 每轮消息全部带上 信息最完整 Token 增长快，成本高
滑动窗口 只保留最近 N 轮 简单稳定 早期关键信息会丢
Token 截断 超过预算就裁剪 能控制成本 可能截断语义
摘要压缩 把历史压成摘要 保留长期背景 摘要质量影响效果
摘要 + 最近 N 轮 长期摘要 + 短期原文 工程上更均衡 实现复杂度更高

RAG 场景下的上下文窗口不只装历史消息，还要装系统 Prompt、用户当前问题、检索到的 chunks，以及预留给模型输出的空间。

详细笔记

大模型每次请求都是“失忆”的

一次 Chat API 请求通常长这样：

模型并不知道数据库里有没有历史记录，也不会自动去读取之前的对话。它能“记住”，只是因为应用层把前面的消息一起传给了它。

所以多轮记忆要从应用架构上设计，而不是指望模型天然保存上下文。

完整历史：最直观但最危险

完整历史就是把从第一轮到当前轮的所有消息都传给模型。

它的优点是简单，信息完整；缺点也明显：轮次越多，Token 越多，成本和延迟都会上升。对 RAG 系统来说更麻烦，因为检索结果也要占上下文。

完整历史适合短对话、调试 Demo，不适合长会话生产系统。

滑动窗口：只保留最近 N 轮

滑动窗口的思路是：只保留最近几轮对话，比如最近 5 轮或 10 轮。

它适合绝大多数普通对话，因为用户当前问题通常和最近上下文关系更强。但它也有一个问题：如果用户一开始说明了身份、偏好、约束条件，后面窗口滑过去后，这些信息就会消失。

Token 截断：从预算角度控制历史

Token 截断不是按轮数，而是按 Token 预算裁剪历史。比如最多只允许历史消息占 4000 Token，超过就从更早的消息开始删除。

它比固定轮数更精细，但要注意不能粗暴截断一条消息中间，否则可能破坏语义。

摘要压缩：把旧历史压成长期记忆

摘要压缩的做法是：当历史消息太长时，用大模型把旧对话总结成一段摘要，后续请求只携带摘要和最近几轮原始消息。

比较稳的摘要内容应该包含：

摘要内容 说明
用户目标 用户到底想完成什么
已确认事实 已经明确过的业务信息
关键约束 时间、格式、范围、权限等限制
未完成事项 后续还要继续处理的问题
用户偏好 表达方式、技术栈、输出风格等

摘要不是越短越好，而是要保留后续推理需要的信息。

推荐策略：摘要 + 最近 N 轮

RAG 场景下的优先级

RAG 问答里，Token 预算要服务于最终回答质量。一般优先级可以这样排：

当前问题和检索结果通常比很久以前的闲聊更重要。否则模型可能“记得聊天”，但忘了引用知识库。

记忆存储怎么选

存储方式 适合场景 注意点
内存 Map 本地 Demo、单机调试 服务重启丢失，不适合集群
Redis 会话短期记忆 要设置 TTL，避免无限增长
MySQL/PostgreSQL 长期审计、历史回放 查询和脱敏要设计好

生产系统通常会组合使用：Redis 保存活跃会话，数据库保存重要会话记录或审计数据。

大模型 API 本身是无状态的，多轮记忆需要业务系统维护历史消息。简单方案可以保留完整历史或滑动窗口，生产环境更适合“摘要压缩 + 最近 N 轮”的混合策略。RAG 场景还要考虑 Token 预算分配，避免历史消息挤占检索上下文，影响答案的事实依据。
