学习导览

MCP 底层使用 JSON-RPC 2.0 作为消息协议。它不是 HTTP，也不是 gRPC，而是一种规定“请求、响应、错误应该长什么样”的 RPC 消息格式。

理解 JSON-RPC 2.0，才能理解 MCP 的 initialize、tools/call、resources/read 这些操作为什么都是统一的消息结构。

知识速览

对象 是否有 id 是否需要响应 用途
Request 有 需要 调用方法并等待结果
Notification 无 不需要 单向通知
Response 有 - 返回 result 或 error
Error 有或 null - 表示协议或调用失败
Batch 多个对象数组 视内容而定 一次发送多个请求/通知

Request Object

字段 含义

method 调用的方法名

Notification

Notification 和 Request 很像，但没有 id，因此服务端不能返回响应。

Error Object

JSON-RPC 预定义了一些协议错误码：

错误码 含义
-32700 Parse error，JSON 解析失败
-32600 Invalid Request，请求对象不合法
-32601 Method not found，方法不存在
-32602 Invalid params，参数不合法
-32603 Internal error，内部错误

业务错误最好不要随便复用协议错误码，可以在业务错误码范围里单独设计。

Batch 批处理

Batch 允许一次发送多个 Request 或 Notification。

如果 Batch 里全是 Notification，服务端不应该返回空数组，而是直接不返回响应。

常见实现错误

错误 后果
忘记 jsonrpc: "2.0" 协议不合法
Notification 返回响应 破坏通知语义
result 和 error 同时存在 响应格式不合法
method 不是字符串 请求不合法
params 类型不对 参数解析失败
命名参数大小写不一致 服务端拿不到参数
业务错误复用协议错误码 排障困难

工程落地建议

协议规范只是基础，真正落地还要补上：

  1. 参数校验。
  2. 超时控制。
  3. 重试策略。
  4. 幂等去重。
  5. 日志和 Trace。
  6. 鉴权和签名。
  7. 版本兼容。
  8. 统一错误码映射。

JSON-RPC 只规定消息格式，不负责解决这些工程问题。

复习抓手

这篇可以记住一句话：

面试回答可以这样说：

MCP 底层使用 JSON-RPC 2.0 作为消息协议。JSON-RPC 定义了 Request、Notification、Response、Error 和 Batch 等对象，Request 通过 id 和 Response 配对，Notification 没有 id 且不需要响应。它只解决消息格式统一问题，不包含鉴权、超时、重试、幂等等工程能力，这些需要业务系统在 MCP 实现外补齐。
