学习导览

前面学习的是 SSE 协议本身，这一篇重点放在 Spring Boot 后端怎么把大模型流式响应转发给前端。

一个常见问题是：为什么前端不直接连大模型 API？

原因有三个：

  1. API Key 不能暴露给浏览器。
  2. 后端需要统一鉴权、限流、日志和审计。
  3. RAG 场景要先做检索、Prompt 组装，再调用模型。

因此后端需要充当流式中间层。

为什么需要后端中间层

SseEmitter 是什么

SseEmitter 是 Spring MVC 里用于 SSE 响应的对象。Controller 返回它以后，请求不会马上结束，后端可以在另一个线程里持续调用 send() 推送事件。

核心方法：

方法 作用
send() 发送一条 SSE 事件
complete() 正常完成
completeWithError() 异常完成
onCompletion() 客户端连接完成回调
onTimeout() 超时回调
onError() 异常回调

生产版要处理什么

超时设置

SseEmitter 默认超时可能不适合长回答，需要根据业务设置。

但超时不能无限大，否则断开的连接可能长期占资源。

心跳保活

很多代理或网关会断开长时间无数据的连接。心跳可以定期发送：

或者发送自定义 heartbeat 事件。

客户端断连检测

用户关闭页面或点击停止生成时，后端要能感知断连，并停止继续调用模型或写数据。

否则模型还在生成，后端还在消费资源。

线程池管理

流式转发可能持续很久，不能把 Tomcat 请求线程一直占住。应该使用独立线程池执行模型调用和转发任务。

Nginx 代理缓冲

如果 Nginx 开了缓冲，后端虽然一直 send，前端可能还是等到缓冲区满或请求结束才看到内容。

常见配置是关闭缓冲：

前端 EventSource 的限制

原生 EventSource 简单好用，但它主要支持 GET 请求，自定义请求头和 POST body 不方便。

如果问答接口需要 POST JSON，可以考虑：

  1. 先 POST 创建任务，再用 EventSource 订阅任务流。
  2. 使用 fetch 读取 ReadableStream。
  3. 封装自定义 SSE 客户端。
